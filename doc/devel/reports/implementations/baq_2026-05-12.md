# BAQ stage — pure-Rust HMM port + per-read driver + rayon stream

**Date:** 2026-05-12
**Implementer:** Claude (Opus 4.7, 1M context)
**Plan:** [`ia/feature_implementation_plans/baq.md`](../../feature_implementation_plans/baq.md)
**Architecture spec:** [`ia/specs/calling_pipeline_architecture.md`](../../specs/calling_pipeline_architecture.md) §"Per-read likelihood quality"

## What was built

The BAQ stage of Stage 1, in three commits:

1. **HMM core** — pure Rust port of htslib's `probaln_glocal`
   ([`src/per_sample_caller/baq/probaln.rs`](../../../src/per_sample_caller/baq/probaln.rs)).
   Reads as htslib's PROBALN_ORIG branch for everything except the
   forward I-state (`xi[1]`) update, which uses the OPTIMIZED-branch
   order so the float-associativity rounding matches the production
   htslib build (the fixtures were generated by that build). Two
   non-obvious traps fixed along the way:
   - `EM = .33333333333` (htslib's literal), not `1.0/3.0` — the
     two `f64` representations differ by ~3.3e-12, which is enough
     to flip Phred rounding at mismatch positions.
   - The OPT-branch's `xi[1] = xm3*y3 + xm4*y4` order vs ORIG's
     `EI * (a + b)` — same math, different floating-point rounding,
     and the htslib fixtures were generated by OPT.

2. **Per-read driver** — `BaqEngine`
   ([`src/per_sample_caller/baq/engine.rs`](../../../src/per_sample_caller/baq/engine.rs))
   owns reusable scratch buffers and wraps the HMM with the CIGAR
   walks, ref-window extension, and `MappedRead` → `PreparedRead`
   conversion that match `sam_prob_realn(b, ref, len, BAQ_APPLY)`.
   Returns either `Capped(PreparedRead)` or `Skipped(reason)` with
   seven reasons mirroring each htslib early-return.

3. **Rayon stream** — `BaqStream`
   ([`src/per_sample_caller/baq/stream.rs`](../../../src/per_sample_caller/baq/stream.rs))
   wraps an upstream `Iterator<Item = Result<MappedRead, …>>`, pulls
   `chunk_size` reads, runs BAQ in parallel via rayon's order-preserving
   `par_iter().map_init(…)`, and yields `Result<PreparedRead, …>`
   serially. Per-reason skip counts roll up into a `total` for the
   per-sample filter summary.

The plan's parameter choices (`d = 1e-3`, `e = 0.1`, `bw = 7`) and
the integration seam (between cram_input and the walker, BAQ runs
*before* the walker sees the read) are unchanged from the plan.

## Caveats / divergences from the plan

- **No end-to-end CRAM → .psp golden file to regenerate.** The plan
  assumed an end-to-end pipeline exists; `main.rs` today is the old
  gVCF merger, and the per-sample CLI is a future slice. The
  integration this commit landed is library-only: `BaqStream` is
  callable from any future CLI that owns a `CramMergedReader` and a
  `RefSeqFetcher`. The plan's "regenerate the golden file" step is
  deferred to whichever commit wires up the CLI.

- **`FilterBucket::BaqRejected` has no call site yet.** The variant
  and the `FilterCounts.baq_rejected: u64` slot exist; the increment
  belongs to the same future CLI that drives `BaqStream`.

- **realn03 is intentionally excluded from the parity test.** Its
  `_exp.sam` is generated by `test_realn -e` (extended BAQ), which
  is out of scope. Its data files remain in `tests/data/baq/` so
  the corpus is easy to extend if extended BAQ ever lands.

- **A small plan-spec discrepancy was found and corrected mid-flight.**
  The architecture spec asserts that htslib and GATK BAQ "converge on
  the same parameters." They don't on `d` — htslib uses `1e-3`, GATK
  uses `1e-4`. The plan picked `1e-3` (samtools production default,
  bcftools calibration parity); the spec correction is a follow-up
  edit not made in this commit.

## Performance

**Hardware:** 12th Gen Intel Core i7-1260P, 16 logical CPUs (4 P-cores
+ 8 E-cores), inside the dev container. Bench profile inherits release
(LTO + 1 codegen unit) with `debug = true` for stack traces.

### Per-read `BaqEngine` cost — `baq_engine_read_length`

Workload: `span = 50_000`, `coverage = 30` → ~1.5 M total bases.
Match-only CIGAR, Q30. One engine reused across all reads in the
iteration (scratch warm). Single-threaded.

| read length | reads / iter | total time | per-read | per-base |
|---:|---:|---:|---:|---:|
|   150 | 10000 | 340 ms |   34 µs | 227 ns |
|   500 |  3000 | 372 ms |  124 µs | 248 ns |
|  1500 |  1000 | 413 ms |  413 µs | 275 ns |
|  5000 |   300 | 412 ms | 1370 µs | 275 ns |

Per-base cost is roughly flat (~230–275 ns), consistent with the
band-limited DP: at `bw = 7` (Match-only, no indel widening), `i_dim`
stays at 39 regardless of `l_query`, so the HMM is genuinely linear
in read length. The small upward creep with `L` is plausibly cache
pressure as the `f`/`b` matrices grow (`(l_query + 1) × i_dim` × 8 B
≈ 47 KB at L=150, 1.5 MB at L=5000).

Extrapolated single-thread time for a 30× WGS sample (~30 M reads at
L=150): **~17 minutes** of pure BAQ.

### `BaqStream` parallel throughput — `baq_stream_chunk_size`

Workload: same 50 kb × 30× fixture at L=150 (typical Illumina); rayon
uses the global default thread pool (16 workers on this host).

| chunk_size | total time | speedup vs 1-thread | reads/sec |
|---:|---:|---:|---:|
|   128 | 103 ms | 3.3× | 97 000 |
|   512 |  93 ms | 3.7× | 108 000 |
|  1024 |  89 ms | 3.8× | 113 000 |
|  4096 |  85 ms | 4.0× | 117 000 |

Speedup tops out at ~4× on a 16-CPU box. On `i7-1260P` this is
plausible: 4 P-cores at full clock + 8 E-cores at ~half throughput
gives an effective parallel budget closer to 8 "P-equivalent" threads,
and the BAQ inner loop hits memory-bandwidth limits well before
saturating that. A larger HPC-class machine would scale further but
this is what a workstation looks like.

The plan's `DEFAULT_BAQ_CHUNK_SIZE = 1024` lands within 5 % of the
optimum (4096) at L=150. **No change to the default required.** The
diminishing returns past 1024 are consistent with chunk-dispatch
overhead becoming negligible relative to per-batch work — and going
much larger would start hurting memory cache locality on real
reads with non-uniform per-read cost.

Extrapolated parallel time for a 30× WGS at L=150: **~4–5 minutes**.

## Conclusions vs plan targets

The plan's only hard performance target was: *"BAQ should not dominate
the per-sample wall clock. If it does (>30 % of total CPU time after
rayon), tune the per-task chunk size."*

A 30× WGS sample on this hardware:

- Pileup walker single-threaded over the same fixture is ~480 ms for
  50 kb at 30× (per the lazy-CIGAR report); extrapolated to a whole
  genome that's ~24 minutes single-thread.
- BAQ parallel on the same workload is ~85 ms at chunk_size=1024;
  extrapolated to the whole genome that's ~4–5 minutes wall clock.

So **BAQ is ~15–20 % of single-thread pileup wall clock**, well below
the 30 % threshold. No chunk-size tuning needed. The default of 1024
stays.

## Out-of-scope follow-ups

- Multi-op CIGAR fixture in the bench (insertions / deletions
  widening `bw`). Match-only is the most predictable; the multi-op
  case is bounded above by what the realn02 / realn03 fixtures
  exercise in the parity test, which is currently clean.
- Long-read parameter validation (htslib auto-switches to `d = 1e-7`
  for `l_qseq > 1000`). The parity fixtures are short-read, so the
  long-read branch is live but untested. Defer until long-read input
  is a real target.
- End-to-end CRAM → `.psp` integration test once the per-sample CLI
  lands. The BaqStream is ready; the missing piece is the CLI plumbing.
- Per-reason skip count surfacing in the user-facing run summary
  (today `BaqSkipCounts` is granular, `FilterCounts.baq_rejected` is
  a flat rollup). Worth doing once a real deployment trips a
  non-trivial skip volume and we want to know which reason dominates.
