## 3. Fixes landed

### 3.1. Drop hom-ref-only sites before the writer — `c7f39a2`

**Symptom found in the dashboard.** 73 % of records in the cohort VCF
had `AC=0` across all 10 samples — non-variant sites with
`AF=0.16;AC=0;AN=20`. The writer had no gate on "this is actually a
variant"; per-sample posteriors said 0/0 but the EM allele-frequency
estimate `p̂` was non-zero, and AF + AC were computed from different
sources, so the record sailed through.

**Fix.** New method `PosteriorRecord::is_variant_call()` —
`best_genotype.iter().any(|&i| i != 0)` (index 0 is the all-REF
genotype by construction of the canonical `genotype_order` table for
every `(ploidy, n_alleles)`). The cohort driver loop skips records
where this returns false, bumping a new `records_dropped_hom_ref`
counter. Surfaced in the run summary.

**Result on the multichrom cohort:** records went 1,358,193 → 365,677
(−73 %), exactly matching the predicted 992,516 AC=0 records the
diagnostic counted. Precision 0.91 % → 3.26 % (+3.6×); recall barely
moved (95.5 % → 94.7 %).

### 3.2. `--min-qual` hygiene filter (default 30) — `d338171`

Standard coarse QUAL floor matching GATK's
`--standard-min-confidence-threshold-for-calling`. Filter at the same
place as the hom-ref-only drop. On the multichrom dataset at this
point in the session this filter dropped ~0 records — our QUAL
formula was inflating from cohort-product math so nothing was below
30. Hygiene shipped anyway; meaning restored by the next fix.

### 3.3. Site QUAL as exact `P(AC = 0 | data)` — `9285578`

**Diagnosis.** Confirmed against primary sources
(`/home/jose/devel/join_per_sample_vcfs/gatk/.../GenotypingEngine.java`,
`AFCalculationResult.java`, GATK issue #1697; FreeBayes paper LaTeX
Eq. probpoly) that GATK and FreeBayes both compute QUAL as a
**single integral over a shared cohort allele-frequency parameter**,
not as a product over per-sample hom-ref posteriors. Our old
`-10·log10(Π_s P(GT_s = hom-ref))` shape made QUAL grow roughly
linearly with cohort size from cohort-product arithmetic alone.

**Replacement.** New `compute_qual_via_exact_af` in
[`src/var_calling/posterior_engine.rs`](../../../src/var_calling/posterior_engine.rs)
— log-domain 1-D convolution over per-sample collapsed-by-alt-count
likelihoods, with a Beta-Binomial(`ploidy·N`, α_alt, α_ref) cohort
prior induced from the same Dirichlet pseudocounts the EM already
uses. Multi-allelic sites are collapsed onto a single "any non-ref"
axis — QUAL answers "is the site polymorphic at all?", not per-ALT.
Standalone Lanczos `ln_gamma` for the Beta-Binomial normaliser
(argument range 0.001..few-thousand, well inside the approximation's
validity).

Three new tests in `posterior_engine.rs`:
- `site_qual_matches_closed_form_for_single_sample` — pins the
  convolution against a hand-computed Beta-Binomial(2, α_alt, α_ref)
  closed form.
- `site_qual_does_not_inflate_when_no_sample_has_strong_alt_evidence`
  — N=1 vs N=20 weak-uniform cohort both give QUAL < 5; the old
  formula grew ~10 phred between them.
- `site_qual_is_large_but_finite_under_extreme_alt_evidence` — pins
  the new "large-finite, no `±∞`" contract.

Plus the old `site_qual_matches_closed_form_on_weak_uniform_cohort`
test (which asserted the buggy joint-product) was removed.

**Result on the multichrom cohort:** records 365,677 → 314,112 (−14 %),
86,291 dropped by the now-meaningful `--min-qual=30`. Precision 3.26 % →
**4.16 %** (+28 %); recall barely moved.

### 3.4. `--min-alt-obs-per-sample` filter (default 2, pre-EM) — `97100d2`

**Diagnosis (per-stage measurement, user-suggested approach).**
Counted records flowing through each pipeline stage:

| Stage | candidates |
|---|---:|
| Positions covered in S0000 pileup | 23,491,865 |
| Positions with any non-REF allele in any sample | **1,396,304 (5.94 %)** |
| Records reaching the EM (≈ same — no filter exists between them) | ~1,392,000 |
| After cohort-AC=0 filter | 400,403 |
| After `--min-qual=30` | **314,112** |

**Alt-AD distribution at the pileup stage** turned out to be the
smoking gun: **89.9 % of candidates are alt_AD = 1** — a single
supporting read on the only carrier. Our pipeline carried every
single-read event into the EM, which dutifully called 0/1 on each
one, and downstream cohort-AC + QUAL filters only caught a fraction
at the back.

Also profiled mean per-read `ln(P_err)` of FP-set ALT evidence to
rule out the parallel "low-BQ" hypothesis — distribution was flat at
ln(P_err) ≈ −3.2 (BQ-equivalent 14, comfortably above any BQ-10
cutoff). Only 2.7 % of FP-set ALT reads sat below BQ 10. The dominant
FP source is the missing alt-AD floor, not BQ.

**Fix.** `filter_map` adapter inserted between `PerGroupMerger` and
`PosteriorEngine` in `drive_cohort_pipeline`. Drops records where no
ALT allele has `max(num_obs across samples) >= N`. Default 2.
Equivalent of GATK's `--min-pruning=2` on the assembly graph but at
a different layer (per-allele max-across-samples rather than
per-De-Bruijn-path).

Saves substantial EM CPU on top of the precision win — ~90 % of
records prune before EM runs.

**Result on the multichrom cohort:**

| metric | before (after fix 3.3) | after N=2 |  Δ  |
|---|---:|---:|---:|
| records emitted | 314,112 | **105,251** | −66.5 % |
| TP (shared) | 17,917 | 17,759 | −0.9 % |
| FP (ours-only) | 296,195 | **87,492** | −70.5 % |
| FN (gatk-only) | 1,525 | 1,683 | +10.4 % |
| precision | 5.7 % | **16.9 %** | +196 % |
| recall | 92.2 % | **91.3 %** | −0.9 pp |
| F1 | 0.107 | **0.285** | +167 % |
| Ts/Tv on FP set | 0.13 | **0.25** | +92 % |

N=3 was also benchmarked; recall dropped 18 pp for a smaller
precision gain. **N=2 is the sweet spot** and is the default.

## 5. Where the remaining FPs are

Even after fix 3.4, the FP set is ~87 K — still ~5× GATK's call set
size, with Ts/Tv 0.25 (well below the biological ~2). Plausible
remaining sources, ranked by what the data hints at:

1. **alt_AD = 2 events** at clustered / repeat sites — sequencing
   errors at the same locus across samples that happen to clear
   N=2 by chance. Investigation: per-chrom + per-region Ts/Tv on
   the residual FP set, look for clustering near repeats / indels.
2. **Indel-flanking misalignments** — pileup-based callers (ours)
   produce phantom SNPs near indels because reads aren't realigned.
   Worth checking the fraction of remaining FPs that sit within
   K bp of an indel.
3. **Strand bias** — pileup tracks fwd/rev counts but doesn't
   filter; GATK's hard filters and VQSR both fold strand bias in.
4. **Real low-frequency variants GATK can't see** — non-zero
   contribution but unlikely to dominate Ts/Tv = 0.25.
