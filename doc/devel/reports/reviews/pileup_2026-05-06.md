## 2. Verdict

**Request-changes.** Two correctness defects in the folding and
mate-overlap paths produce wrong scalar counts in records whose
REF span exceeds 1, which is exactly the case where the walker is
doing real work (deletions, compound haplotypes). The spec has been
followed for the simple SNP path but diverges from the spec on the
deletion / wide-record path, on the mate-overlap tie-break, and on
the indel-mate-overlap "single observation" rule. Fixes are
localized but each requires an added regression test before
re-approval.

## 4. Open Questions & Assumptions

1. **Indel-mate-overlap collapses to one observation (B2).** The
   spec is explicit: "Both mates report the same indel at the same
   anchor — Treat as one observation, not two." The current code
   zeroes the loser's BQ but still folds it as a separate
   observation. This review assumes the spec's text is the binding
   contract; if the project has decided to relax it (e.g. for
   parity with the SNP-overlap rule), the spec needs an update —
   *not* the test.

2. **`find_overlapping` early-break (M1).** The comment at
   [open_record.rs:172-181](../../src/per_sample_caller/pileup/open_record.rs#L172-L181)
   self-acknowledges that the early break is "for correctness …
   we just terminate the search" while saying "in practice we'd
   still need to walk a couple back to be safe". This review
   treats that as a real bug; if there is an unwritten invariant
   that no two open records can have overlapping footprints in
   this codebase, please document it and convert the run-time
   check into an `assert!` on the invariant.

3. **REF accumulation policy at widened records.** The spec's Step
   6 (REF bucket maintenance) describes folding REF reads "once,"
   at record open / widen time. Today's implementation folds at
   every walker step where the read has an event landing inside an
   already-open record. Findings B1 and the proposed regression
   test in §8 assume "once" is the binding rule.

## 5. Top 3 Priorities

1. **B1 — REF (and any allele) is folded once per walker step
   inside the open record's footprint, multiplying scalars by the
   record's `ref_span`.** Confirmed by experiment; reproducer runs
   in <1s. Worst affected: deletion records and any compound /
   widened records, exactly the cases the walker exists to
   handle.
2. **B2 — Mate-overlap on indels does not collapse to a single
   observation.** Same record, same fold loop; loser's BQ is
   zeroed but `num_obs` is still incremented. Spec's
   "indel-overlap = one observation" rule unmet.
3. **M2 — Mate-overlap BQ tie-break uses `alignment_start`,
   not `is_first_mate`.** Breaks for any properly-paired read
   with the first mate at the higher coordinate (typical for
   reverse-strand mates in Illumina paired-end data).

## 6. Findings

### Blocker

#### B1: [open_record.rs:434-484](../../src/per_sample_caller/pileup/open_record.rs#L434-L484), [walker.rs:200-265](../../src/per_sample_caller/pileup/walker.rs#L200-L265) — Reads inside an open wide record are re-folded at every walker step

- **Confidence:** High
- **Assumptions:** Spec §"Step 4-6" of
  [pileup_walker.md](../specs/pileup_walker.md) — "fold every
  contributor whose events overlap the record's footprint exactly
  once". Open question 3.
- **Problem:**
  `process_position` adds a record's anchor key to `affected`
  whenever any event at the current `walker_pos` lands inside it
  (via `find_overlapping`), regardless of whether the record was
  just opened, widened, or already covered the event with no
  change. Then the inner loop folds *every* contributor that has
  events in the record's footprint into the record. For a read
  whose CIGAR-walk produces Match events at every position inside
  the record (i.e. any pure-Match read covering the record), the
  contributor is in the contributors list at every walker step
  inside the record's footprint, and the record is in `affected`
  at every such step (because the Match event keeps finding it).
  Result: the read is folded once per walker step within the
  record's footprint, and *all five scalars* (`num_obs`, `q_sum`,
  `fwd`, `placed_left`, `placed_start`) are multiplied by the
  record's `ref_span`.

  Confirmed by experiment. A scratch test was added (and removed
  after the experiment) with two reads: r1 = `5M` at start 1 (pure
  REF over `ACGTA`), r2 = `1M3D1M` at start 2 (3-bp deletion).
  Expected REF.num_obs at the deletion record at pos 2 (span 4) = 1
  (one ref-spanning read). Observed:

  ```
  DEBUG: anchor record alleles:
   [AlleleObservation { seq: [67, 71, 84, 65],            // REF "CGTA"
                        scalars: FiveScalars { num_obs: 4,
                                               q_sum: -12.0,
                                               fwd: 4,
                                               placed_left: 4,
                                               placed_start: 0 },
                        chain_slots: [0] },
    AlleleObservation { seq: [67],                         // DEL "C"
                        scalars: FiveScalars { num_obs: 1, ... },
                        ... } ]
  assertion `left == right` failed: REF should have 1 obs from r1 only;
  got 4
  ```

  REF was folded 4 times (once per walker step in [2, 6)). All five
  scalars are 4× too large.

- **Why it matters:**
  Five-scalar correctness is the single thing Stage 1 must
  deliver. The downstream likelihood is:
  `Σ max(ln_BQ, ln_MQ)` over supporting reads
  ([calling_pipeline_architecture.md §"The five per-allele scalars"](../specs/calling_pipeline_architecture.md))
  — counting the same read N times produces a `prodQout` term that
  is N× the truth. `placed_left` / `placed_start` / `fwd` are
  similarly wrong. The bug is silent: SNP records (span 1) are
  unaffected, so the existing test suite does not catch it; only
  records that span more than one base are wrong, which is exactly
  the deletion / compound-event case the walker exists to handle.

- **Suggested fix:**
  Two parts.

  **Part A — restrict `affected` to records that were just
  *created* or *widened* at this walker step.** A record that
  already covers the event's footprint with no change does not
  need re-folding. Track creation / widening events explicitly:

  ```rust
  // in process_position:
  let mut affected: Vec<u32> = Vec::new();
  for contrib in contributors {
      for ev in &contrib.events_at_pos {
          let event_start = ev.anchor_pos();
          let event_end = event_start + ev.footprint_span();
          let mut touched = false;
          let key = if let Some(k) = open.find_overlapping(event_start, event_end) {
              let cur_end = open.records.get(&k)
                  .expect("just located").footprint_end_exclusive();
              if event_end > cur_end {
                  open.widen(k, event_end, fasta)?;
                  touched = true;          // widened → re-fold needed
              }
              k
          } else {
              open.open_new(chrom_id, event_start, ev.footprint_span(), fasta)?.pos;
              touched = true;              // newly opened → fold needed
              event_start
          };
          if touched && !affected.contains(&key) {
              affected.push(key);
          }
      }
  }
  ```

  **Part B — fold against *every* active read whose events overlap
  the affected record's footprint, not only the contributors at
  the current walker step.** Active reads admitted in earlier
  walker steps are still in the active set; they need to be folded
  exactly once per record they overlap. The natural shape: pass
  the active set into `process_position` and iterate it instead of
  `contributors` for the Step 4-6 fold. To avoid double-folding a
  read that has already been folded at a prior open/widen, track
  per-record "already-folded read_ids" inside `OpenPileupRecord`
  and only fold reads not yet in that set. (This is also what
  makes mate-overlap finding 1's tie-break correct — see M2.)

  After the fix, the missing test in §8
  (`deletion_record_does_not_double_count_ref_reads`) becomes the
  permanent regression guard.

#### B2: [walker.rs:382-460](../../src/per_sample_caller/pileup/walker.rs#L382-L460), [open_record.rs:447-484](../../src/per_sample_caller/pileup/open_record.rs#L447-L484) — Mate overlap on indels increments `num_obs` twice

- **Confidence:** High
- **Assumptions:** Spec
  [pileup_walker.md](../specs/pileup_walker.md) §"Mate overlap
  on indels" — "Both mates report the same indel at the same
  anchor — Treat as one observation, not two". Open question 1.
- **Problem:**
  `resolve_mate_overlap_at_pos` zeroes the loser's BQ on every
  event (Match, Insertion, Deletion) but does not drop the loser
  from the contributor list. When both mates of a pair report an
  Insertion at the same anchor, both contributors fold into the
  open record at the same allele bucket; both increment
  `num_obs`. The spec says one observation, not two. Same applies
  to a same-anchor Deletion seen by both mates.

  Match-Match overlap is correctly handled (the spec there *does*
  call for "still counts as 1 obs, with `q_sum` contribution
  zeroed"); the bug is specifically in the indel branch.

- **Why it matters:**
  Indel evidence-counting is a load-bearing input for downstream
  variant calling. Counting one observation as two doubles the
  apparent indel coverage from any overlapping mate pair, biasing
  `num_obs` upward at any anchor where mate insert size puts the
  indel inside both reads' coverage windows. Realistic for any
  insert size shorter than `2 × read_length`, which is the typical
  Illumina paired-end regime.

- **Suggested fix:**
  Distinguish indel-overlap from match-overlap inside
  `resolve_mate_overlap_at_pos`:

  ```rust
  // For an indel-shaped event present on both contributors at the
  // same anchor: drop the loser's event entirely (do not just zero
  // bq_proxy) so num_obs only increments once at fold time. For a
  // Match-Match pair: keep the loser, zero its bq.
  for indices in by_slot.values().filter(|v| v.len() >= 2) {
      let (a, b) = (indices[0], indices[1]);
      let (loser, kind) = decide_loser(&contributors[a], &contributors[b]);
      match kind {
          OverlapKind::MatchMatch => zero_match_bq(&mut contributors[loser]),
          OverlapKind::SameIndel  => drop_event_from(&mut contributors[loser]),
      }
  }
  ```

  `drop_event_from` removes the indel event from
  `events_at_pos` and `full_window_events`, so the fold step skips
  it. Both contributors still share the slot id, so phase-chain
  bookkeeping is unaffected.

  Also add `is_first_mate` to `ReadContribution` (see M2) so the
  tie-break is correct.

### Major

#### M1: [open_record.rs:161-183](../../src/per_sample_caller/pileup/open_record.rs#L161-L183) — `find_overlapping` early break is unsound when records have heterogeneous spans

- **Confidence:** High
- **Assumptions:** Spec [pileup_walker.md](../specs/pileup_walker.md)
  §"Step 3" — `open_records.range(..=event_anchor_pos)` followed by a
  scan. Open question 2.
- **Problem:**
  After locating the largest `Q ≤ event_start`, the scan iterates
  *backward* through records. The exit condition is

  ```rust
  if rec.footprint_end_exclusive() <= event_start {
      break;
  }
  ```

  This terminates at the first record whose footprint ends at or
  before `event_start`. But a record with a *narrow* footprint
  sitting between `event_start` and a *wide* earlier record (e.g.
  a SNP record at position 40 with span 1, and an earlier
  deletion record at position 5 with span 50) will cause the
  scan to break at the narrow record without ever inspecting the
  wide one — even though the wide record's footprint reaches into
  `event_start`.

  The author has flagged this in a comment ("we'd still need to
  walk a couple back to be safe, but for correctness we just
  terminate"), so the bug is known but not yet fixed.

- **Why it matters:**
  Two open records with overlapping footprints is not pathological
  in this design — a long deletion opening a wide record can stay
  open while shorter records open and close around it. When this
  happens, an event that should have folded into the wide record
  instead opens a fresh record, breaking the per-record allele
  uniqueness invariant ("every allele in the record spans the same
  reference stretch"). The downstream encoder cannot recover from
  this — it serialises whatever the walker produced.

- **Suggested fix:**
  Replace the early break with a search bounded by `MAX_RECORD_SPAN`
  (the maximum reach any single record could have):

  ```rust
  pub fn find_overlapping(&self, event_start: u32, event_end: u32) -> Option<u32> {
      let lo = event_start.saturating_sub(MAX_RECORD_SPAN);
      for (&q, rec) in self.records.range(lo..=event_start).rev() {
          if rec.footprint_end_exclusive() > event_start && q < event_end {
              return Some(q);
          }
      }
      None
  }
  ```

  This walks at most `MAX_RECORD_SPAN` keys in the worst case
  but is still `O(log n + k)` where `k` is the number of records
  whose anchor is in `[lo, event_start]`. In practice `k` is small
  because open records close eagerly. Add a regression test with
  three records: a wide deletion, a narrow SNP between, and an
  event whose footprint falls inside the deletion's range.

#### M2: [walker.rs:425-436](../../src/per_sample_caller/pileup/walker.rs#L425-L436) — Mate-overlap BQ tie-break uses `alignment_start`, not `is_first_mate`

- **Confidence:** High
- **Assumptions:** Spec
  [pileup_walker.md](../specs/pileup_walker.md) §"Tie-breaking
  and disagreement" — "BQ tie: prefer mate 1 (flag `0x40`,
  `is_first_mate` set)".
- **Problem:**
  The tie branch in `resolve_mate_overlap_at_pos`:

  ```rust
  std::cmp::Ordering::Equal => {
      if contributors[a].alignment_start <= contributors[b].alignment_start {
          b
      } else {
          a
      }
  }
  ```

  drops whichever contributor has the larger `alignment_start`. The
  spec says drop the non-first-mate, regardless of position.
  These coincide only for the common case where the first mate has
  the smaller start. They diverge when the first mate is the
  reverse-strand half of the pair (typical for `R2` placed leftward
  in some library prep / aligner configurations) or when both
  mates start at the same coordinate (tie on start → spec wants
  first-mate, code picks `b` arbitrarily).

  Root cause: `ReadContribution` does not carry `is_first_mate`,
  so the walker has no choice but to substitute `alignment_start`.

- **Why it matters:**
  The wrong tie-break flips which mate's BQ contribution is kept
  vs zeroed, changing `q_sum` (the principal quality scalar) at
  any equal-BQ overlap position. Determinism across runs is
  preserved (same input → same output), but the output disagrees
  with the spec.

- **Suggested fix:**
  Add the field to the contribution and use it:

  ```rust
  pub struct ReadContribution {
      // ... existing fields ...
      pub is_first_mate: bool,
  }
  // populate in walker.rs::process_position:
  contributors.push(ReadContribution {
      // ... existing fields ...
      is_first_mate: active.read.is_first_mate,
  });
  // tie branch:
  std::cmp::Ordering::Equal => {
      if contributors[a].is_first_mate { b } else { a }
  }
  ```

#### M3: [walker.rs:251-262](../../src/per_sample_caller/pileup/walker.rs#L251-L262) — `record_widen_events` counter conflates widen and open

- **Confidence:** High
- **Assumptions:** Spec [pileup_walker.md](../specs/pileup_walker.md)
  §"Run-summary counters" — `record_widen_events: times an open
  record's REF span widened`.
- **Problem:**
  The counter is computed as

  ```rust
  let span_before: u32 = self.open.iter().map(|(_, r)| r.ref_span()).sum();
  // ... process_position ...
  let span_after: u32 = self.open.iter().map(|(_, r)| r.ref_span()).sum();
  if span_after > span_before {
      self.summary.record_widen_events += 1;
  }
  ```

  Opening a fresh record at this step also increases the sum (a
  new record contributes its full `ref_span` to the new total), so
  the counter increments on opens as well as widens. The metric is
  therefore "any record-table mass increase," not "widen events" —
  unobservably wrong for QA.

- **Why it matters:**
  Runtime metrics are how operators detect surprise (long-deletion
  storms, alignments forcing wide windows). A wrong metric points
  away from real causes.

- **Suggested fix:**
  Plumb a real "widened" signal out of
  `OpenPileupRecordTable::widen` (return a `bool` or a small
  enum), increment the counter only when widening actually
  happened, and report opens separately under a new
  `record_open_events` counter (or fold both into the run summary
  if that is what operators want).

#### M4: [open_record.rs:551-564](../../src/per_sample_caller/pileup/open_record.rs#L551-L564), [walker.rs:267-283](../../src/per_sample_caller/pileup/walker.rs#L267-L283) — Lifecycle marks always attach to the *first* drained record, dropping marks when no record drains

- **Confidence:** High
- **Assumptions:** Spec
  [pileup_walker.md](../specs/pileup_walker.md) §"Stamping
  records" — "When the walker closes a record at `pos`, it drains
  the allocator's `new_marks` and `expired_marks` since the last
  emitted record into `PileupRecord.new_chains` and
  `expired_chains`".
- **Problem:**
  `close_aged_records` calls `drain_lifecycle_marks` *only* if
  there are aged records to close (`if aged.is_empty() { return
  Ok(()); }` at
  [walker.rs:269-271](../../src/per_sample_caller/pileup/walker.rs#L269-L271)).
  When a slot is allocated *and* released between two emissions —
  for example, a single read whose entire footprint falls in a
  walker step where no record happens to age out — the marks
  accumulate in `new_marks` / `expired_marks` until the next
  emission, where they are stamped on whatever record happens to
  emit then. If no further record ever closes (e.g. end-of-input
  with the chromosome already finished) the marks are flushed via
  `flush_chromosome`, which still attaches them to the *first*
  drained record:

  ```rust
  if let Some(first) = records.first_mut() {
      first.new_chains = new_chains;
      first.expired_chains = expired_chains;
  }
  ```

  A subtler issue: the `stamp_lifecycle_marks` comment claims
  "Stage 5's recv loop applies them before reading allele
  observations, so attaching to the first record is enough" — but
  the spec says marks are per-emission ("drains the allocator's
  marks since the last emitted record"). When more than one
  record drains at once, the *second* and later records lose any
  attached state from this emission window: their `new_chains` /
  `expired_chains` is unconditionally `Vec::new()`. If Stage 5
  does its merge under the assumption that each record carries
  its own delta, this is wrong.

- **Why it matters:**
  Phase-chain identity across positions is the only mechanism by
  which Stage 5 can detect compound haplotypes. A lost
  `expired_chains` mark or a mis-attached `new_chains` mark is a
  silently corrupted graph the merger has no way to validate.
  Manifests as a Stage-5 join error far from its cause.

- **Suggested fix:**
  Either (a) the spec needs to be amended to say "marks since the
  last emission attach to the *next* emission's first record"
  unambiguously, *and* the implementation should drain regardless
  of `aged.is_empty()` so accumulated marks aren't held forever;
  or (b) marks attach per-record (each record carries the marks
  that became visible since the previous *record*, not the
  previous emission). Pick one, document it, and add a test in
  which a slot's lifetime spans a no-emission walker step. The
  early-return at
  [walker.rs:269](../../src/per_sample_caller/pileup/walker.rs#L269)
  must be replaced with an unconditional drain so deferred marks
  reach Stage 2.

#### M5: [walker.rs:396-460](../../src/per_sample_caller/pileup/walker.rs#L396-L460) — `resolve_mate_overlap_at_pos` only inspects adjacent index pairs

- **Confidence:** Medium
- **Assumptions:** When more than two contributors share a
  `chain_slot_id` at the same walker_pos, the comparison should
  look at *every* pair, not just adjacent ones from a `windows(2)`
  iterator.
- **Problem:**
  The pair iteration is `for window in indices.windows(2)`. With
  `indices = [a, b, c]`, this compares (a,b) and (b,c) but not
  (a,c). For a typical mate pair (`indices.len() == 2`), the
  behaviour is correct. For three or more contributors sharing a
  slot — which can happen if supplementary or split alignments
  ever survive the upstream filter, or if a future change loosens
  the filter — the comparison is incomplete and the wrong
  contributors may be selected as winners.
- **Why it matters:**
  Defensive at the moment (the spec says paired mates only share
  slots, and supplementary reads are filtered upstream), but
  brittle: any future change that admits a third reader of the
  same slot silently mis-resolves. Robust loops over all pairs
  are cheap when N is tiny.
- **Suggested fix:**
  Replace `indices.windows(2)` with the all-pairs combinations:

  ```rust
  for i in 0..indices.len() {
      for j in (i + 1)..indices.len() {
          let (a, b) = (indices[i], indices[j]);
          // ... existing comparison ...
      }
  }
  ```

  Add a `debug_assert!(indices.len() <= 2,
      "spec invariant: only mate pairs share a slot")` to make
  the implicit assumption explicit, so a future filter-change
  that violates it shows up in tests rather than in production.

### Minor

#### Mi1: [open_record.rs:307](../../src/per_sample_caller/pileup/open_record.rs#L307) — `checked_sub(record_pos).unwrap_or(0)` masks a precondition violation

- **Confidence:** High
- **Problem:**
  `apply_events_to_ref` is documented as "Events are assumed to be
  inside the record's footprint." If `ev.anchor_pos() <
  record_pos`, the `checked_sub` returns `None` and the code
  silently treats `offset = 0`, fabricating an event at the
  record's anchor position. Any call site that violates the
  precondition gets a wrong haplotype string instead of an error.
- **Why it matters:**
  Hides a real correctness contract behind a "saturating" pattern.
  Clippy's hint to use `saturating_sub` would also hide it. The
  right shape is to require the precondition (with a `debug_assert!`)
  and propagate the bug in the test environment.
- **Suggested fix:**

  ```rust
  debug_assert!(
      ev.anchor_pos() >= record_pos,
      "event anchor {} below record_pos {}",
      ev.anchor_pos(), record_pos,
  );
  let offset = ev.anchor_pos() - record_pos;
  ```

  The clippy lint stops firing because the explicit subtraction is
  no longer wrapped in an `unwrap_or`.

#### Mi2: [walker.rs:285-294](../../src/per_sample_caller/pileup/walker.rs#L285-L294) — `expire_passed_reads`'s `walker_pos == 0` guard is dead code

- **Confidence:** High
- **Problem:**
  `WalkerState::new` initialises `walker_pos = 1`, and
  `set_chrom_if_needed` resets it to 1 on chromosome change.
  There is no path that sets `walker_pos = 0`. The early-return
  guard is dead.
- **Why it matters:**
  Defensive code that never fires is noise that obscures the real
  control flow. A future refactor that does set `walker_pos = 0`
  for some reason inherits a silent skip with no failure to point
  at the misuse.
- **Suggested fix:**
  Drop the guard:

  ```rust
  fn expire_passed_reads(&mut self) -> Result<(), WalkerError> {
      self.active.expire_passed(self.walker_pos, &mut self.slots)
  }
  ```

  If defensive coverage is wanted, replace with a `debug_assert!(self.walker_pos > 0)`.

#### Mi3: [walker.rs:343-347](../../src/per_sample_caller/pileup/walker.rs#L343-L347) — `flush_chromosome` accepts an unused `_fasta`

- **Confidence:** High
- **Problem:**

  surface still has to wonder why it's there.
- **Why it matters:**

#### Mi4: [walker.rs:111-122](../../src/per_sample_caller/pileup/walker.rs#L111-L122) — `RunSummary::from_slot_counters` takes `Self` by value

- **Confidence:** Medium
- **Problem:**
  The method ignores `self_`'s slot fields (`slot_allocations`,
  `slot_high_water`, `mate_lookup_evictions` always come from the
  argument), and returns a fresh struct. A more honest shape is a
  free function or a `RunSummary::with_slot_counters(self,
  counters) -> Self` consuming `self`. The trailing-underscore
  parameter name (`self_`) is an idiom workaround for the keyword
  collision; a regular `Self` method (`fn into_with_counters(self,
  …)`) reads better.
- **Why it matters:**
  Naming and ergonomic — the function is correct, just awkward.
- **Suggested fix:**
  Convert into a method on `RunSummary`:

  ```rust
  impl RunSummary {
      fn into_with_slot_counters(mut self, slot: SlotAllocatorCounters) -> Self {
          self.slot_allocations = slot.slot_allocations;
          self.slot_high_water = slot.slot_high_water;
          self.mate_lookup_evictions = slot.mate_lookup_evictions;
          self
      }
  }
  // call site:
  self.summary.into_with_slot_counters(self.slots.counters())
  ```

#### Mi5: [mod.rs:106-131](../../src/per_sample_caller/pileup/mod.rs#L106-L131) — `FiveScalars::zero` duplicates `Default::default`

- **Confidence:** High
- **Problem:**
  `FiveScalars` derives `Default`. The custom `zero()` method
  delegates to `Self::default()`. Two ways to spell the same
  thing.
- **Why it matters:**
  Names should be canonical: a reader hitting `Self::zero()` has
  to verify it isn't doing something different from
  `Self::default()` and what semantics "zero" carries beyond the
  derived defaults. With nothing extra here, prefer
  `Default::default()` everywhere or commit to `zero()` and
  document why it exists separately.
- **Suggested fix:**
  Drop `zero()`. Replace its two use sites
  ([mod.rs:127-131](../../src/per_sample_caller/pileup/mod.rs#L127-L131),
  [open_record.rs:32](../../src/per_sample_caller/pileup/open_record.rs#L32))
  with `FiveScalars::default()`.

#### Mi6: [open_record.rs:251](../../src/per_sample_caller/pileup/open_record.rs#L251) — `widen` always appends `extra_bases`, including for INS-bearing alleles

- **Confidence:** Medium
- **Assumptions:** Spec
  [pileup_walker.md](../specs/pileup_walker.md) §"Step 4.2"
  worked example: when REF widens, an existing INS allele's
  inserted bases stay where they are and the new REF bases are
  appended *past the insertion's anchor*.
- **Problem:**
  The current implementation appends `extra_bases` to *every*
  allele's `seq` regardless of allele type:

  ```rust
  for allele in &mut rec.alleles {
      allele.seq.extend_from_slice(&extra_bases);
  }
  ```

  For a SNP/MNP allele whose `seq.len() == ref_seq.len()` before
  widening, this is correct (the new REF bases haven't been
  modified by the read). For a DEL allele whose `seq.len() <
  ref_seq.len()` (anchor only) the spec says to keep the deletion
  length and append the new REF bases past the deletion — also
  what the code does. For an INS allele whose `seq.len() >
  ref_seq.len()` because it carries inserted bases, the code
  appends the new REF bases at the *end of the inserted run*,
  effectively placing the new REF bases inside the inserted
  segment in some cases, depending on where the insertion's anchor
  sat in the old REF. Worked example: REF=`AT`, INS allele=`A_XX_T`
  (insertion of XX at anchor offset 0); widen REF to `ATC`,
  appending `C` to the INS allele yields `A_XX_T_C` — happens to be
  right because the insertion anchor was at offset 0, so the
  remainder of REF (`T`) sits to the right of the insertion in
  the allele. But if a future change (compound INS+SNP, or an INS
  whose anchor is not at offset 0) widens, the simple "append"
  rewrite is no longer well-defined.
- **Why it matters:**
  Today's tests don't exercise compound allele widening; the
  shortcut works. Subtle to detect when the assumption breaks.
- **Suggested fix:**
  Either (a) document and assert the invariant ("INS allele's
  inserted bases sit at the right end of the allele seq under the
  current REF span"), or (b) rewrite alleles by re-applying the
  read's events to the new REF (the same machinery
  `apply_events_to_ref` already implements). Option (b) costs
  redundant FASTA work but eliminates the invariant. Choose (a) if
  the runtime cost matters; add an assertion test that fails
  loudly when the invariant breaks.

## 9. What's Good

- **Spec-driven layout matches the implementation cleanly.**
  [pileup_walker.md](../specs/pileup_walker.md) §"Module
  layout" lists eight files; the implementation uses six and the
  difference (no `mate_overlap.rs` / `five_scalars.rs`) is
  justified by the small size of each. The mapping is easy to
  follow.
- **`OpenPileupRecord::ref_span()` derives from `ref_seq.len()`**
  rather than carrying a separate field. Matches the spec's
  invariant and removes a class of "two fields out of sync"
  bugs by construction.
- **`PendingMate::seen_at` + `MATE_LOOKUP_WINDOW` eviction
  bumps a counter on eviction**
  ([slot_allocator.rs:268-289](../../src/per_sample_caller/pileup/slot_allocator.rs#L268-L289)).
  The metric reaches the run summary, so an operator can see when
  this is firing.
- **`drain_lifecycle_marks` suppresses slots present in both
  lists**
  ([slot_allocator.rs:200-233](../../src/per_sample_caller/pileup/slot_allocator.rs#L200-L233))
  — the spec's `[QUESTION]` is resolved cleanly and exercised
  by `drain_lifecycle_marks_suppresses_slots_present_in_both`.
- **`MockFasta` is a tight stub that owns its bytes
  in-memory and translates 1-based → 0-based at the trait
  boundary**
  ([tests.rs:31-65](../../src/per_sample_caller/pileup/tests.rs#L31-L65)).
  Useful pattern for any future I/O abstraction the project adds.
