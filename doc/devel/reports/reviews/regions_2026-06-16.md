## 2. Verdict

**Approve-with-changes.** No Blockers. The feature is structurally sound, idiomatic, and well-tested at the example level; the gaps cluster in **failure transparency** (silent empty output, lost error context), **refactor-safety** of the per-region counter folds, one **correctness margin** in the per-region worker teardown, and a handful of test gaps. Two findings (M2, M3) are **recurrences of prior-review Majors that were never applied** (the prior review predates this code's current shape, so they carried forward).

## 4. Open questions and assumptions

1. **(affects M1)** What is the intended contract for a `--regions` BED that parses but selects nothing (empty file, all-comment, or zero overlap with the reference)? Three options: hard error (`EmptyRegionSet`), warn-and-continue, or document silent-empty as valid. The fix for M1 depends on this choice — it cannot be inferred from the code (today it is silent-empty).
2. **(affects M5/Mi11)** Is `MappedReadSource` / `with_stage1_chain` intended as a *swappable* reader/stage seam (forward-looking abstraction) or a single-consumer helper? The answer decides whether to minimize its visibility (Mi10/Mi11) or keep it `pub` and documented.

## 5. Top 3 priorities

1. **M4** — [stage1_pipeline.rs:311-355](../../../../src/pop_var_caller/stage1_pipeline.rs#L311-L355): per-region worker `JoinHandle`s are dropped, not joined; a worker panic mid-fold can drop a sequence number and let the walker finish over a *truncated* read stream. Only `thread::scope`'s panic re-raise (before the atomic rename) currently prevents wrong output — an implicit, undocumented margin. The one correctness-adjacent finding.
2. **M1** — silent empty `.psp`/VCF on a non-selecting `--regions` BED (convergent: reliability, defaults, smells). User-facing wrong-feeling result with exit 0; resolve the contract (Q1) then error/warn.
3. **M3** — the three per-region counter-fold `merge()`s are not field-addition-safe (prior M1, still unfixed); `RunSummary::merge`'s per-field `max` semantic makes a copy-paste `+=` on a future field a silent bug.

## 6. Findings

### Major

**M1: [src/var_calling/pipeline.rs:282-323](../../../../src/var_calling/pipeline.rs#L282-L323) + [src/pop_var_caller/cli.rs:319-401](../../../../src/pop_var_caller/cli.rs#L319-L401) — A `--regions` BED that selects nothing produces a silent valid-but-empty output (exit 0)**
- **Categories:** reliability, defaults, smells (convergent)
- **Confidence:** High. An empty/all-comment BED → empty `RegionSet` → the cohort `schedule` is empty (header-only VCF) and the pileup `for region in region_set.iter()` loop runs zero times (valid empty `.psp`, all-zero run summary). Pinned by `regions.rs` tests `empty_bed_yields_empty_region_set` (L660) and `restrict_intervals_empty_regions_yields_empty` (L788) — the *empty set* is tested, but not the *user-facing silence*.
- **Fix:** resolve Q1, then either return a typed `EmptyRegionSet` error or emit an `eprintln!` warning at both call sites (the project uses `eprintln!`, no `tracing` dep). Add regression tests asserting the chosen behavior at both the pileup and cohort entry points.

**M2: [src/regions.rs:129,296-297](../../../../src/regions.rs#L296-L297) — `BedError::Io` drops the path + line number and flattens its `io::Error` with no `#[source]` (prior review M3, still unfixed)**
- **Category:** errors
- **Confidence:** High. It is the only line-originating variant that doesn't carry `line_number` (in scope at the `?` site), it is mechanism-named not operation-named, and `{0}` Display flattening drops the `io::Error` from the chain.
- **Fix:** rename to `ReadLine { line_number, #[source] source: io::Error }`, attach via `.map_err` at the read site. (`BedError::Open` has the same flatten shape but carries `path` — Nit.)

**M3: [src/bam/alignment_input.rs:141](../../../../src/bam/alignment_input.rs#L141), [src/pileup/per_sample/baq_stream.rs:76](../../../../src/pileup/per_sample/baq_stream.rs#L76), [src/pileup/walker/driver.rs:260](../../../../src/pileup/walker/driver.rs#L260) — the three per-region counter-fold `merge()`s are not field-addition-safe (prior review M1, still unfixed)**
- **Category:** refactor_safety
- **Confidence:** High. `FilterCounts::merge` (folded at cli.rs:436), `BaqSkipCounts::merge` (cli.rs:441), `RunSummary::merge` (cli.rs:435) each hand-list `self.field += other.field`. A new field is silently omitted from the per-region fold → wrong totals. `RunSummary::merge` is worst: `active_reads_high_water` uses a per-field `max`, so a copy-paste `+=` on a new field is doubly wrong. `#[non_exhaustive]` gives no *in-crate* protection.
- **Fix:** destructure `*other` (and `*self` where needed) with no `..`, so a new field is a compile error in the fold. (The `StageCounts`→`FilterCounts` one-off at stage1_pipeline.rs:244 has the same shape — Minor.)

**M4: [src/pop_var_caller/stage1_pipeline.rs:311-355](../../../../src/pop_var_caller/stage1_pipeline.rs#L311-L355) — per-region worker `JoinHandle`s are dropped, not joined; a worker panic can silently truncate the read stream**
- **Category:** unsafe_concurrency
- **Confidence:** Medium (Needs verification of the panic-timing window). In `run_pipelined`, only the producer is explicitly joined. If a worker panics after popping a `RawPacket` but before sending its `ResultPacket` (e.g. the `.expect` per-worker fetcher build at read_pipeline.rs:177), that sequence number is lost; `ReorderReadIter` then yields end-of-stream **with a gap** once senders drop, and the walker closure completes over a truncated stream. The only thing preventing wrong output escaping is `thread::scope` re-raising the panic and aborting before the `.tmp`→output rename — an implicit, undocumented safety margin.
- **Fix:** collect and join the worker handles before trusting the walker's clean exit (mirroring the producer join and the cohort driver's documented pattern), or at minimum document the invariant that `thread::scope`'s re-raise is load-bearing here.

**M5: [src/var_calling/pipeline.rs:431-595](../../../../src/var_calling/pipeline.rs#L431-L595) — the staged and inline producer branches duplicate a ~25-line interval-walk that must be kept in sync**
- **Category:** smells
- **Confidence:** High. Each branch carries a near-identical `dust_mask_for` + `begin_interval` + fetch-closure + `plan_chunk` inner loop. This is the exact shape of the prior staged-path multi-interval regression (two copies drifted). 
- **Fix:** extract a shared `drive_schedule(..., ship: impl FnMut(ChunkPlan))` helper both branches call, so the interval-walk logic lives once.

**M6: [tests/pileup_cli_integration.rs](../../../../tests/pileup_cli_integration.rs) — no test asserts `--regions` output is byte-identical across `--threads` (the project's stated regression gate)**
- **Category:** reliability
- **Confidence:** High. Every `--regions` integration test uses `threads: None` (= 4 → the staged path only); the inline path (`< STAGED_MIN_THREADS`) is never exercised under `--regions`, so the cross-thread byte-identity gate is unverified for this feature.
- **Fix:** add a test running the same `--regions` BED at `--threads 1` and `--threads 4` and asserting byte-identical `.psp` (mirror the existing cross-thread oracle used elsewhere).

## 9. What's good

- **`regions.rs` coordinate naming is exemplary** — `bed_start`/`start_1based`/`region_end_excl`, with the 0-based-half-open → 1-based-inclusive conversion documented at the boundary ([regions.rs:240-241](../../../../src/regions.rs#L240-L241)).
- **Determinism is designed-in** — `sort_and_merge` uses a total-order `(chrom_id, start, end)` key (instability unobservable) and `argmin_head` breaks ties to the lowest input index ([segment_merge.rs:190](../../../../src/bam/segment_merge.rs#L190)).
- **The error enum is correctly `#[non_exhaustive]`** with tuple-match (not length-then-index) parsing in `regions.rs`.
- **The pooled-reader concurrency teardown is sound and documented** — `drop(reorder)` before `producer.join()` avoids the errors-without-draining deadlock; poison recovery via `into_inner()` (verified by unsafe_concurrency, building on the 2026-06-16 segment-reader review).
- **Both cleanup commits on this branch are behavior-identical** — the qname move and the `Stage1Outputs` field removal were traced to be safe (extras).
