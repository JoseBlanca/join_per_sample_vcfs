## 6. Findings

### Major

**M1: `src/sample_summary/mod.rs:38` — on-disk `.psp` key renamed without a schema-version bump.**
**Categories:** refactor_safety, defaults (convergent). **Confidence:** High.
`CoverageByGcHistogram` derives serde with `#[serde(rename_all = "kebab-case")]`, so renaming the field `n_tiles`→`n_positions` changes the persisted TOML key `n-tiles`→`n-positions`. The field is required (no `#[serde(default)]`), so a stale v2 document fails with a missing-field `ParseToml` error while still being *labelled* version 2 — the version guard the module documents as the clean-rejection mechanism never fires. Not a wrong-results bug (serde errors rather than zero-defaulting), but a diagnosability/format-contract regression against the module's own "every bump is breaking" policy.
**Fix applied:** bumped `SAMPLE_SUMMARY_VERSION` 2→3 with a v3 history entry documenting the rename and the "stale v2 rejected at the TOML layer" behavior.

### Minor

**Mi1: `src/sample_summary/mod.rs:303` — stale `n-tiles` wording in the `callable_positions` validation error.**
**Categories:** reliability, refactor_safety, defaults, smells (convergent, 4×). **Confidence:** High.
The rename reached the field and format args but not the human-readable string literal (`"{} < n-tiles {} …"`), which now mislabels the interpolated `n_positions` value with a retired key name. The existing regression test asserted only the `field` tag, so it did not catch the drift.
**Fix applied:** string now reads `n-positions`; surrounding "folded tile" comments reworded to "folded (window) sample / position".

### Nits (applied where in-scope)

- `mod.rs:294-298`, test doc comments `mod.rs:544-570` — "folded tile" vocabulary describing what is now `n_positions`; reworded.
- `examples/dump_sample_summary.rs:110` — format-string column padding kept its old `n_tiles` width. Left as-is: the example is pre-existing-broken and out of scope (see §7).

## 7. Out of scope observations

- **`examples/dump_sample_summary.rs` does not compile** — calls `SampleSummaryAccumulators::observe_record` / `new(…)` / `finish()` against an API that has since changed to `stage_record` / `new(coverage, het)` / `finish::<W>`. Pre-dates M6 (M5's `clippy` used default targets, which skip examples). Suggested follow-up: a separate small PR to update or retire this debug example.
- **Mixed `n_positions` / `n_skipped_tiles` vocabulary** (`mod.rs:105-117`) — the task deliberately scoped only `n_tiles`. On the live `SlidingWindowCoverageAccumulator` path `n_skipped_tiles` is hard-wired to `0`, so the surviving "tile" field is always zero there. Suggested follow-up: either rename `n_skipped_tiles` (finishing the vocabulary pass) or add a legacy-note; do not let the split state become permanent.
- **Legacy tiled `CoverageByGcAccumulator`** (`coverage.rs`) is now used only by `examples/paralog_score_parity.rs`, not the production pileup path (which uses `SlidingWindowCoverageAccumulator`). Worth a future decision on whether to keep it as a parity oracle or retire it. Its internal `n_tiles` counter genuinely counts tiles and correctly feeds the histogram's `n_positions` field (documented at `coverage.rs:123-126`).

## 9. What's good

- Struct-literal exhaustiveness (no `Default` impl, no `..`) made the compiler flag every `CoverageByGcHistogram` construction site, so the field rename could not silently default anywhere (`refactor_safety`).
- The window-spill removal reused only shared codec primitives (`put_f32`/`write_frame`/`Decoder`), so deleting the window format left the record-spill codec — and thus VCF byte-identity — provably untouched (`reliability`, `refactor_safety`).
- `reference_base_matches` was relocated to sit directly above its sole caller in `write_pass.rs`, with its unit test, rather than left exported from a vestigial module (`module_structure`).
- The rename fails **loud** (missing-field parse error), not silent-wrong, because the field is required with no `#[serde(default)]` (`defaults`).
