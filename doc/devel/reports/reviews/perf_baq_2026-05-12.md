## 2. Verdict

**Profile first.** The bench gives wall-time per L sweep and per chunk size but does not attribute time to any specific line inside `probaln.rs` or `engine.rs`. Section 3 lists the measurements that promote the section 5 findings from pattern-match to evidence-backed.

The candidates exist and most have well-defined experiments; once the flamegraph lands, the highest-leverage three are likely **(L1 + L2 + L3) per-read allocations**, **(L10 + L11) rayon load imbalance / pipeline-serial bottleneck** as the explanation for the ~4× ceiling on 16 cores, and **(L7 + L8 + L9) inner-loop bounds-check / encoding work** — but pick the lever that the profile actually points at.

## 3. Measurement plan

Run these in order; each unblocks downstream findings.

1. **Sampling profile of `baq_engine_read_length/150`** — the per-read engine hot path. `[profile.bench]` already inherits release with `debug = true` (Cargo.toml:11-13) so this is one command:
   ```
   cargo flamegraph --bench baq_perf -- baq_engine_read_length/150 --profile-time 30
   ```
   or `samply record cargo bench --bench baq_perf -- baq_engine_read_length/150 --profile-time 30`. Threshold: any single self-time site above ~15% becomes a Hot-path finding and dictates which of L1–L9 to chase. Pasted top-10 self-time sites belong appended to [ia/reports/implementations/baq_2026-05-12.md](../reports/implementations/baq_2026-05-12.md).

2. **Allocation profile of the same bench, gated on a feature flag**:
   ```
   cargo bench --features dhat-heap --bench baq_perf -- baq_engine_read_length/150
   ```
   The project already has `dhat-heap` wired (Cargo.toml:30-34). Report which allocation sites have per-read frequency (`apply_baq_cap`'s `Vec<u8>`, the qname `Arc<str>`, `cigar.clone()`, `seq.clone()`, and the per-chunk Vecs in `BaqStream`). Threshold: a fix is worth merging if it removes a per-read allocation site from the DHAT top-10.

3. **Length-skewed engine bench**. Today `build_mapped_reads` builds a single-length fixture and `ConstRefFetcher` returns `vec![b'A'; len]`. Both hide cost:
   - **Cache bias:** an all-A ref + all-A query is L1-resident and trivializes the emission branch in `probaln.rs:205`. Add a mixed-base reference fixture (deterministic A/C/G/T/N mix) and a mixed-query fixture. The current flat ~230–275 ns/base across L may be a microbench artefact.
   - **Load-imbalance bias:** today's chunk is uniform-L. Add a fixture mixing 90 % short (150 bp) and 10 % long (5 kb) reads to test the rayon length-mix splitting hypothesis (L10).

4. **Saved-baseline + realistic-fetcher chunk-size bench** at the production `ChromBoundaryRefFetcher` over a real FASTA fixture (cold and warm page cache). Required to lock or revisit `DEFAULT_BAQ_CHUNK_SIZE = 1024` and to give any I/O finding (L12, L13) measurable hot-path evidence. Save with `cargo bench --save-baseline baq-2026-05-12` so subsequent experiments compare against today's numbers.

5. **Per-phase wall-time inside `BaqStream::refill_batch`** for the pipeline-serialization hypothesis (L11). Instrument `drain`, `par`, and `collect+bump` separately with `std::time::Instant`. Threshold for the channel-pipeline rewrite: `T_drain + T_walker_consume > 15% T_par` on the realistic-fetcher bench.

## 4. Build / toolchain configuration

These are independent of code-level work and can land before section 5.

- **Set `target-cpu` (Likely).** The crate ships under `lto = "fat"` and `codegen-units = 1` (Cargo.toml:6-9) but no `.cargo/config.toml` exists, so the HMM `f64` MAC chains in [probaln.rs:206-212](../../src/per_sample_caller/baq/probaln.rs#L206-L212), [probaln.rs:285-288](../../src/per_sample_caller/baq/probaln.rs#L285-L288), and [probaln.rs:336-348](../../src/per_sample_caller/baq/probaln.rs#L336-L348) compile against the `x86-64` SSE2 baseline rather than AVX2/FMA. Workstation/HPC x86_64 is the named target. The diff:
  ```toml
  # .cargo/config.toml (new file)
  [build]
  rustflags = ["-C", "target-cpu=x86-64-v3"]
  ```
  Measurement: rerun `baq_engine_read_length/*` with `--save-baseline target-cpu-v3`; cross-check with `cargo asm` that the inner loops emit `vfmadd*pd` / `vmulpd` / `vaddpd`. Worth merging if geomean of `baq_engine_read_length` drops ≥ 10%. **Parity:** AVX2 vfmadd vs scalar mul+add changes rounding; the existing parity tests will detect any divergence. If they break, fall back to `target-feature=+avx2,+fma` (which still allows scalar pipelined FMA) or drop to `x86-64-v2` (`+sse4.2`).

- **Global allocator override (Likely).** The streaming path allocates per chunk (`stream.rs`) and per read (`engine.rs`); rayon's work-stealing reassigns reads across threads so cross-thread free/reuse is the dominant traffic shape. `mimalloc` / `jemallocator` have per-thread caches that satisfy small `Vec<u8>` / `Vec<CigarOp>` allocations without an arena lock. Add behind a feature flag:
  ```toml
  [dependencies]
  mimalloc = { version = "0.1", default-features = false, optional = true }

  [features]
  mimalloc = ["dep:mimalloc"]
  ```
  Wire `#[global_allocator]` in `src/main.rs`. Measurement: `cargo bench --features mimalloc --bench baq_perf -- baq_stream_chunk_size --save-baseline mimalloc`. Worth merging if `/1024` drops ≥ 5% wall time, or if the gap between single-thread `baq_engine_read_length/150` and the per-read cost inside `baq_stream_chunk_size` narrows. Confirm with DHAT that allocation **count** does not change (only the latency should).

- **(Note) `panic = "abort"`** is not set in `[profile.release]`. Removes per-frame unwinding metadata and may help LLVM inline a few `probaln.rs` callees. Trivial change; small expected effect.
  ```toml
  [profile.release]
  panic = "abort"
  ```

- **(Note) `rust-toolchain.toml` not pinned.** The 2026-05-12 measurements are not reproducible without recording the exact rustc version they were taken on. Pin the channel via `rust-toolchain.toml`; cheap one-file change.

- **(Speculative — Build) PGO.** A representative-workload PGO build is the highest-leverage *build* lever after LTO + target-cpu + allocator. Defer until those three are absorbed.

## 5. Code-level findings

### Likely

- `src/per_sample_caller/baq/engine.rs:242` — **L1: `apply_baq_cap` allocates a fresh `Vec<u8>` per read**
- **Confidence:** High
- **Hot-path evidence:** pattern-match only — `BaqEngine::process` is the success-path tail on every non-skipped read (100M–300M times per WGS sample per the prompt).
- **Pattern matched:** "allocations belong outside hot loops" — the engine already owns `state`/`q`/`tref`/`tquery` as reusable buffers, but `apply_baq_cap` returns a fresh `Vec<u8>` via `qual.to_vec()` ([engine.rs:242](../../src/per_sample_caller/baq/engine.rs#L242)).
- **Mechanism:** one heap allocation (sized to the read length, ~100–300 B) per `Capped` outcome. The Vec moves into `PreparedRead.bq_baq` and is dropped downstream when the walker releases the read. An engine-owned scratch + `mem::take` swap turns N per-process allocations into a single allocation that survives the engine's lifetime.
- **Measurement plan:** DHAT in step 2 of section 3. The fix is worth merging if `<alloc::vec::Vec<T> as Clone>::clone` (or the matching `apply_baq_cap` site) drops by one per read in the "blocks by site" view, and `baq_engine_read_length/{150,500}` does not regress.
- **Complexity cost:** one extra `Vec<u8>` field on `BaqEngine`; a `std::mem::take` plus `Vec::clear` + `extend_from_slice`. No new lifetime, no `unsafe`.
- **Suggested experiment / fix:**
  ```rust
  pub struct BaqEngine { /* ... */ bq_buf: Vec<u8> }

  fn apply_baq_cap_into(out: &mut Vec<u8>, /* same args */) {
      out.clear();
      out.extend_from_slice(qual);
      // ...write into out[i] as today...
  }

  // In `process`, after a successful probaln_glocal:
  apply_baq_cap_into(&mut self.bq_buf, /* ... */);
  let bq_baq = std::mem::take(&mut self.bq_buf);
  ```

- `src/per_sample_caller/baq/engine.rs:291` — **L2: qname conversion allocates twice per read (`from_utf8_lossy` + `Arc::<str>::from`)**
- **Confidence:** High
- **Hot-path evidence:** pattern-match only — same per-read frequency as L1.
- **Pattern matched:** "watch for accidental `Cow::Owned`" + "clones are evidence".
- **Mechanism:** `String::from_utf8_lossy(&read.qname)` returns `Cow::Borrowed` in the happy path (SAM qnames are ASCII per spec) and allocates a fresh `String` in the bad-byte path. The `Arc::<str>::from(&str)` then always allocates an Arc header + a copy of the bytes. Two heap allocations per read worst-case, one happy-case. A fast path through `std::str::from_utf8` cuts the happy path to one allocation total.
- **Measurement plan:** same DHAT as L1.
- **Complexity cost:** one helper function, one branch for the (very rare) non-ASCII fallback. No new dependency.
- **Suggested experiment / fix:**
  ```rust
  fn qname_to_arc(qname: &[u8]) -> Arc<str> {
      match std::str::from_utf8(qname) {
          Ok(s) => Arc::<str>::from(s),
          Err(_) => Arc::<str>::from(String::from_utf8_lossy(qname).as_ref()),
      }
  }
  ```

- `src/per_sample_caller/baq/engine.rs:299-300` — **L3: `read.cigar.clone()` and `read.seq.clone()` per read**
- **Confidence:** Medium
- **Hot-path evidence:** pattern-match only — runs on every `Capped` outcome.
- **Pattern matched:** "clones are evidence, not noise" — each `.clone()` on a `Vec` needs a justification.
- **Mechanism:** `MappedRead` is owned by the rayon chunk and dropped right after `process`. The engine borrows `&MappedRead` and clones two `Vec`s into the `PreparedRead`. If `process` consumed `MappedRead` by value, both Vecs could be moved into the `PreparedRead` for the cost of three pointer copies each. `MappedRead.qname` then needs `qname_to_arc(&read.qname)` *before* the move (see L2). The note on [cram_input.rs:85-88](../../src/per_sample_caller/cram_input.rs#L85-L88) explicitly motivates owning-vs-borrowing for the same reason.
- **Measurement plan:** same DHAT — the `Vec<T>::clone` site count for `cigar`/`seq` should drop by two per read; rerun `baq_engine_read_length/150` and check the absolute change.
- **Complexity cost:** API change — `process(read: MappedRead, ...)` instead of `&MappedRead`. `BaqStream::refill_batch` switches `par_iter()` → `into_par_iter()`. No `unsafe`, no new lifetime; the existing skip-cases (which return without producing a `PreparedRead`) just drop the now-owned `MappedRead`.
- **Suggested experiment / fix:**
  ```rust
  pub fn process(&mut self, read: MappedRead, ref_fetcher: &dyn RefSeqFetcher) -> BaqOutcome
  { /* ... */ }

  fn mapped_to_prepared(read: MappedRead, bq_baq: Vec<u8>) -> PreparedRead {
      PreparedRead {
          chrom_id: read.ref_id as u32,
          // ...
          cigar: read.cigar,    // moved
          seq: read.seq,        // moved
          bq_baq,
          qname: qname_to_arc(&read.qname),
          // ...
      }
  }

  // stream.rs::refill_batch:
  let outcomes: Vec<BaqOutcome> = chunk
      .into_par_iter()
      .map_init(|| BaqEngine::new(cfg), |engine, read| engine.process(read, fetcher))
      .collect();
  ```

- `src/per_sample_caller/baq/stream.rs:127-141` — **L4: per-chunk `Vec<BaqOutcome>` + `Vec<PreparedRead>` materialise on every refill**
- **Confidence:** Medium
- **Hot-path evidence:** pattern-match only — one allocation pair per chunk; chunk pass amortised over `chunk_size = 1024` reads, so the per-read share is small but the alloc-count line item is non-zero in DHAT.
- **Pattern matched:** "`.collect::<Vec<_>>()` followed by another iteration is wasted work" — `outcomes` collects, then immediately drains into `prepared`. Both Vecs could be engine-owned scratch.
- **Mechanism:** rayon needs `outcomes` to be collected to preserve input order, but the buffer can be reused across `refill_batch` calls via `rayon::iter::collect_into_vec` (it accepts a `&mut Vec<T>` and clears it). `prepared` (the surviving subset) can become a `Vec<PreparedRead>` + cursor field on `BaqStream`, so its backing buffer stays put across refills rather than leaving every chunk via `IntoIter`.
- **Measurement plan:** `cargo bench --bench baq_perf -- baq_stream_chunk_size`. Threshold: no regression on the points the bench already reports; the gain is in DHAT (two allocation sites removed from the per-chunk count).
- **Complexity cost:** two new `Vec` fields on `BaqStream`, swap `current_batch: IntoIter<PreparedRead>` for `Vec<PreparedRead>` + a `current_batch_idx: usize` cursor. The Iterator impl gets slightly more state but the borrow shape stays the same.
- **Suggested experiment / fix:**
  ```rust
  pub struct BaqStream<'a, R, F> {
      /* ... existing ... */
      outcomes_buf: Vec<BaqOutcome>,
      current_batch: Vec<PreparedRead>,
      current_idx: usize,
  }
  // refill_batch uses chunk.into_par_iter().map_init(...).collect_into_vec(&mut self.outcomes_buf);
  ```

- `src/per_sample_caller/baq/scratch.rs:23-26` — **L5: per-worker 1 KB `q2p` lookup duplicated across rayon workers**
- **Confidence:** Medium
- **Hot-path evidence:** pattern-match only — `Scratch::new` is invoked once per worker per `par_iter` (i.e. once per chunk, per worker), and inside it `10f32.powf(-(i as f32) / 10.0)` runs 256 times. On a 16-worker box that is 16 × 256 = 4096 `powf` calls per chunk; at ~200M reads / 1024 reads per chunk that is ~200k chunks → ~819M `powf` calls per sample, plus 16 × 1 KB of cache footprint duplication.
- **Pattern matched:** read-only lookup tables belong in a `&'static`.
- **Mechanism:** `q2p` is immutable, deterministic, and identical across all workers. Lifting it to `static Q2P: LazyLock<[f32; 256]>` (or a build-script-generated const literal) eliminates the per-worker copy and the per-chunk re-initialisation. The per-call cost is small individually; the value is mostly in the reduced cache pressure and the cleaner `Scratch` shape.
- **Measurement plan:** micro-bench `BaqEngine::new(cfg)` (1000× iter) before and after; rerun `baq_stream_chunk_size/{128,1024,4096}`. Worth merging if `BaqEngine::new` drops below ~1 µs and the stream bench moves measurably above noise on small `chunk_size`.
- **Complexity cost:** one `static` + a helper that `Scratch::new` consults. No `unsafe`, no new dependency.
- **Suggested experiment / fix:**
  ```rust
  // scratch.rs
  use std::sync::LazyLock;

  static Q2P: LazyLock<[f32; 256]> = LazyLock::new(|| {
      let mut t = [0f32; 256];
      for (i, slot) in t.iter_mut().enumerate() {
          *slot = 10f32.powf(-(i as f32) / 10.0);
      }
      t
  });

  pub(super) struct Scratch {
      pub(super) f: Vec<f64>, pub(super) b: Vec<f64>,
      pub(super) s: Vec<f64>, pub(super) qual: Vec<f32>,
      // q2p removed
  }
  // probaln.rs: let q2p: &[f32; 256] = &*Q2P;
  ```

- `src/per_sample_caller/baq/probaln.rs:203-218` — **L6: forward / backward inner loops index `Vec<f64>` through `Index<usize>` — per-cell bounds checks**
- **Confidence:** Medium
- **Hot-path evidence:** pattern-match only — three nested `k in beg..=end` loops do 3 writes + 7 reads each, on ~150 inner iterations per row × `l_query` rows × per read.
- **Pattern matched:** "bounds checks: avoid by structure, not by `unsafe`" — the `assert!` hoist sub-rule.
- **Mechanism:** every `f[base_i + u]` access goes through `<Vec<f64> as Index<usize>>::index`, which is bounds-checked. The compiler cannot prove `base_i + u + 2 < f.len()` because `u` comes from `set_u(bw, i, k)` whose `(i - bw).max(0)` defeats simple range analysis. A single hoisted `assert!(base_i + i_dim <= f.len())` above each `k`-loop is the structural proof LLVM needs; in-band `u`s are < `i_dim` by construction (`bw2 * 3 + 6 = i_dim`).
- **Measurement plan:** `cargo bench --bench baq_perf -- baq_engine_read_length/150` before/after. Confirm with `cargo asm` (or `--emit=asm`) that `panic_bounds_check` calls disappear from the inner loop bodies.
- **Complexity cost:** three to four `assert!` statements above each k-loop. No new types, no `unsafe`.
- **Suggested experiment / fix:**
  ```rust
  // Above each `for k in beg..=end {` block (lines 203, 278, 340):
  assert!(base_i + i_dim <= f.len());
  assert!(base_i1 + i_dim <= f.len());
  let u_max = set_u(bw, i, end) + 2;
  assert!(u_max < i_dim);
  ```
  **Parity:** bounds-check elision changes no floating-point arithmetic. Parity tests should pass unchanged.

- `src/per_sample_caller/baq/engine.rs:113` — **L7: `encode_base` is a 5-way `match` in a per-base hot path**
- **Confidence:** Medium
- **Hot-path evidence:** pattern-match only — runs `xe - xb` + `l_qseq` times per read (~300+ times per Illumina read), 100M–300M reads/sample.
- **Pattern matched:** "match on bytes for ASCII parsers" + "autovectorization needs the compiler's confidence" — a `[u8; 256]` const lookup replaces a tree of compares with a straight-line gather LLVM can vectorize.
- **Mechanism:** LLVM may already lower the 5-arm match to a jump table, but a 256-byte LUT is unambiguous and lets `Vec::extend(iter.map(encode_base))` autovectorize cleanly. The function is not on the FP parity surface — `encode_base` only feeds the `u8` `tref`/`tquery` buffers, not the `f64` DP.
- **Measurement plan:** `cargo bench --bench baq_perf -- baq_engine_read_length`. The win is largest on the L=150 slot where the encode pass is a larger share of per-read cost. Verify with `cargo asm` that the encode loop emits `vpgatherdd` / `vpshufb`-shape SIMD on AVX2.
- **Complexity cost:** one ~12-line const block. Net simpler at the call site.
- **Suggested experiment / fix:**
  ```rust
  static ENCODE_BASE: [u8; 256] = {
      let mut t = [4u8; 256];
      t[b'A' as usize] = 0; t[b'a' as usize] = 0;
      t[b'C' as usize] = 1; t[b'c' as usize] = 1;
      t[b'G' as usize] = 2; t[b'g' as usize] = 2;
      t[b'T' as usize] = 3; t[b't' as usize] = 3;
      t
  };
  pub(super) fn encode_base(b: u8) -> u8 { ENCODE_BASE[b as usize] }
  ```

- `src/per_sample_caller/baq/probaln.rs:203-210` — **L8: `set_u` called four times per inner k-iter; `(i - bw).max(0)` is loop-invariant**
- **Confidence:** Medium
- **Hot-path evidence:** pattern-match only — same hot loops as L6.
- **Pattern matched:** "avoid recomputing in tight loops; hoist invariants".
- **Mechanism:** `set_u(bw, i, k) = ((k - (i - bw).max(0) + 1) * 3) as usize`. For fixed `i` and `bw`, `x = (i - bw).max(0)` is constant across the inner `k` loop. The four `set_u(bw, i, k)`, `set_u(bw, i-1, k-1)`, `set_u(bw, i-1, k)`, `set_u(bw, i, k-1)` calls all reduce to `(k + const) * 3` with two row-level constants. LLVM may already hoist this — `cargo asm` first.
- **Measurement plan:** `cargo asm` to confirm whether `.max(0)` (lowered to `cmov` or a branch) appears inside the inner loop. If it does, hoist; if not, close.
- **Complexity cost:** two extra row-local bindings + four manual offset computations replacing the `set_u` calls. Slightly less readable; the doc comment migrates.
- **Suggested experiment / fix:**
  ```rust
  let x_i   = (i     - bw).max(0);
  let x_im1 = (i - 1 - bw).max(0);
  for k in beg..=end {
      let u   = ((k     - x_i   + 1) * 3) as usize;
      let v11 = ((k - 1 - x_im1 + 1) * 3) as usize;
      let v10 = ((k     - x_im1 + 1) * 3) as usize;
      let v01 = ((k - 1 - x_i   + 1) * 3) as usize;
      /* ... */
  }
  ```
  **Parity:** identical `usize` offsets — no FP changes.

- `src/per_sample_caller/baq/probaln.rs:330-354` — **L9: MAP pass loads the dead D-state per cell (AoS where SoA fits)**
- **Confidence:** Medium
- **Hot-path evidence:** pattern-match only — three sweeps over the same `f`/`b`; at L≥1500 the matrices exceed L2 and the MAP pass refetches.
- **Pattern matched:** "SoA when each loop reads exactly one field" — here the MAP pass reads two of three sub-fields per cell.
- **Mechanism:** `Scratch.f` / `Scratch.b` interleave M/I/D as three consecutive `f64` per cell (24 B). Forward and backward read all three states per cell (AoS is right for them). MAP at [probaln.rs:336-348](../../src/per_sample_caller/baq/probaln.rs#L336-L348) only touches M (`+0`) and I (`+1`); the D lane (`+2`) is fetched into L1 with the cache line and never used. Splitting D into a parallel `Vec<f64>` (so each matrix becomes a packed M/I plane + a separate D plane) cuts MAP-pass bytes-read-per-cell from 24 B to 16 B per matrix.
- **Measurement plan:** profile-first (section 3 step 1) to confirm MAP is a meaningful share of self-time. Then implement the split behind a feature flag and re-run `baq_engine_read_length/{1500,5000}`. Threshold: MAP-only time drops by more than the cache-bound share predicted by `perf stat -e LLC-load-misses` on the pre-change build.
- **Complexity cost:** **High** — two storage shapes for DP cells, parallel `set_u`-style indexing for the D lane, rewrite of `resize_for`, boundary init/consume paths, and the htslib-parity comments. **Parity risk: real.** The fixed-stride AoS layout is implicit in many of the f-index expressions; splitting risks an off-by-cell bug that the parity test would catch. The change is gated on the parity-test fixtures (`tests.rs::parity_realn0[12]`) passing without modification.
- **Suggested experiment / fix:** prototype only after the profile confirms MAP is hot. Sketch in [data_layout.md L1](../../tmp/perf_review_2026-05-12_baq/data_layout.md).

- `src/per_sample_caller/baq/stream.rs:127-133` — **L10: rayon default splitting over a length-mixed chunk leaves a heavy tail**
- **Confidence:** Medium
- **Hot-path evidence:** pattern-match only on the **shape**; the prompt notes ~4× speedup on 16 logical CPUs. The plan attributes it to i7-1260P P+E asymmetry and memory bandwidth; this finding is the additional load-imbalance hypothesis.
- **Pattern matched:** "rayon's default splitting assumes roughly uniform per-item cost".
- **Mechanism:** `chunk: Vec<MappedRead>` is coordinate-sorted, so a mixed-length CRAM can produce a chunk with a >30× per-item cost spread between 150 bp and 5 kb reads. Rayon's default depth-balanced binary split has no cost awareness; the worker that gets the long tail finishes last while the rest idle. No `with_min_len` tuning, no cost-aware partitioning.
- **Measurement plan:** add a length-skewed chunk fixture (90% short + 10% long) per section 3 step 3 and rerun `baq_stream_chunk_size`. Compare against (a) sort-by-descending-length pre-split + final reorder, and (b) `.with_min_len(64)` alone. Capture per-worker wall time with `rayon::current_thread_index()` + thread-local `Duration` accumulators. Worth merging if the worst/best-thread ratio drops materially on skewed fixtures.
- **Complexity cost:** O(n log n) sort per refill (negligible for n = 1024 relative to per-read BAQ cost) + a `(index, MappedRead)` zip and a final outcomes-by-index sort. Sketch in [concurrency.md L1](../../tmp/perf_review_2026-05-12_baq/concurrency.md).

- `src/per_sample_caller/baq/stream.rs:103-141` — **L11: producer (CRAM drain) and consumer (walker) are serialised around the parallel section — no pipeline overlap**
- **Confidence:** Medium
- **Hot-path evidence:** pattern-match only on the **shape** — same ~4× speedup. `refill_batch` is strictly sequential: drain upstream → `par_iter` → drain into `current_batch` → return to walker. The walker then drains `current_batch` to exhaustion before the next `refill_batch`. Workers are idle during phases 1 and 3 (and during the walker phase between refills).
- **Pattern matched:** "channels: bounded by default" / pipeline parallelism — when stages cannot overlap, total time is `T_drain + T_par + T_walker_consume`, not `max(...)`.
- **Mechanism:** the walker is single-threaded; the CRAM iterator is single-threaded. A bounded-channel pipeline (CRAM thread → `crossbeam_channel::bounded(2)` → BAQ workers → bounded(2) → walker) would let all three stages overlap.
- **Measurement plan:** section 3 step 5. Threshold for the channel-pipeline rewrite: `T_drain + T_walker_consume > 15% T_par` on the realistic-fetcher bench (section 3 step 4).
- **Complexity cost:** **High** — two bounded channels, a dedicated CRAM-fetch thread, `JoinHandle` cleanup on iterator drop, error propagation as a channel message. Breaks the "single-threaded `next()` is straightforward" property the [stream.rs:1-7](../../src/per_sample_caller/baq/stream.rs#L1-L7) doc currently advertises.

- `src/per_sample_caller/baq/engine.rs:107` — **L12: per-read `RefSeqFetcher::fetch` for an ~150 B window; consider a chunk-scoped batch fetch**
- **Confidence:** Medium
- **Hot-path evidence:** pattern-match only — the bench's `ConstRefFetcher` "hides any real I/O cost" per the prompt. The production `ChromBoundaryRefFetcher` (out of scope) wraps `noodles_fasta::Repository`, which has per-call overhead (vtable, FASTA-repo lookup, possible internal locking).
- **Pattern matched:** "buffering and chunked reads matter the same way" — when the inner step is small (~150 B) and called 100M–300M times, per-call overhead dominates the actual byte cost.
- **Mechanism:** within one `refill_batch` the reads are coordinate-sorted and overlap heavily on the reference. A `fetch_span(chrom, start, end)` issued once per (chrom, run) in the chunk, with per-read slicing into the returned buffer, collapses 1024 fetches into one per chrom-run.
- **Measurement plan:** section 3 step 4 — realistic-fetcher bench, cold + warm page cache; `perf stat -e syscalls:sys_enter_read,page-faults`. Threshold: ≥ 15% wall time on the warm-cache run, or ≥ 5× drop in `read` syscalls per chunk.
- **Complexity cost:** new trait method (`fetch_span`) with a default impl that loops over `fetch` — no caller is forced to migrate. `refill_batch` grows a pre-pass that computes per-read `(xb, xe)`, takes chunk-wide min/max per chrom, calls `fetch_span` once per run, then runs the rayon pass with read-only slices. Lifetime: the buffer is local to `refill_batch` and outlives the `map_init` closure.
- **Suggested experiment / fix:** prototype as a new `process_with_ref(read, ref_bytes, xb)` on `BaqEngine` without changing the trait; promote the trait method only if the prototype clears the threshold.

- `src/per_sample_caller/baq/engine.rs:84,107` — **L13: BAQ and the pileup walker each call `fetch` per record over overlapping windows**
- **Confidence:** Medium
- **Hot-path evidence:** pattern-match only — the call duplication is from the prompt's "the two callers (BAQ stage and walker) share the same fetcher" + the BAQ-side single `fetch` at [engine.rs:107](../../src/per_sample_caller/baq/engine.rs#L107).
- **Pattern matched:** "do the expensive lookup once, hand the result downstream".
- **Mechanism:** `mapped_to_prepared` does not forward the fetched ref slice; the walker's per-record callback reissues a `fetch` over an overlapping (often strictly-contained) window. 2× the fetcher cost per read on the BAQ→walker path.
- **Measurement plan:** `perf record -F 999 -g` over a full BAQ + walker pass; check time inside `ChromBoundaryRefFetcher::fetch`. Threshold: ≥ 10% wall-time reduction across the combined stages, or > 50% reduction in fetch calls per read.
- **Complexity cost:** **High and cross-cutting.** Either widens `PreparedRead` with an owned `Vec<u8>` (allocation per read, fights `simplify`'s reuse goal) or with a `(chunk_buf: Arc<[u8]>, offset, len)` triple (lifetime / Arc plumbing). Ripples into the walker's struct definition — out of this commit's scope. Best handled jointly with L12 in a follow-on design pass.

### Speculative

- `src/per_sample_caller/baq/probaln.rs:117-119` — **S1: three-sweep f/b traversal exhausts L3 at long L; pass fusion could reduce bandwidth**
- **Confidence:** Low
- **Mechanism:** forward + backward + MAP each touch the full `f` and `b` matrices. With 16 workers each carrying ~3 MB of live `f`+`b` at L=5000, the aggregate ~48 MB working set exceeds the i7-1260P's 18 MB L3. Fusing the MAP pass into backward would keep only the current row of `f`/`b` live (`i_dim * 16 B` ≈ 0.6 KB at bw=7), L1-resident.
- **Parity risk:** fusing two passes changes the read order of `f` cells, which can change the value of `state[i]` if the MAP argmax sees a different version of `f[i][k]` (currently always the post-renormalisation value). The htslib parity is a deciding constraint; the change is unlikely to land without redefining parity as tolerance-based rather than byte-for-byte.
- **Measurement plan:** `perf stat -e LLC-load-misses,mem_load_retired.l3_miss` on `baq_engine_read_length/5000` to confirm the L3-miss hypothesis. Prototype only after the section 3 profile points at it.

- `src/per_sample_caller/baq/engine.rs:84` — **S2: `&dyn RefSeqFetcher` forces a vtable hop per read; monomorphise**
- **Confidence:** Low
- **Mechanism:** the call site re-erases the `F: RefSeqFetcher + Sync` generic `BaqStream` carries. Per-call overhead is small but real; the inliner cannot specialise across the boundary.
- **Measurement plan:** swap signature to `process<F: RefSeqFetcher + ?Sized>(&mut self, read: ..., fetcher: &F)` and rerun the realistic-fetcher bench. Threshold: ≥ 2% wall time. Bench delta is small **today** and shrinks further after L12, since per-call population drops by ~1024×.
- **Complexity cost:** minor — already matches `BaqStream`'s generic shape. Small codegen-bloat risk if many fetcher types eventually exist (unlikely).

- `src/per_sample_caller/baq/probaln.rs:299-301` — **S3: per-row backward rescale loop should autovectorize but isn't verified**
- **Confidence:** Low
- **Mechanism:** the loop is straight-line, bounded, contiguous slice, no branches — textbook AVX2 vectorizable. Bounds checks may inhibit. Restructuring as `b[start..=end].iter_mut().for_each(|x| *x *= scale)` may help the compiler.
- **Measurement plan:** `cargo asm` first; if the loop already emits `vmulpd`, close.
- **Parity:** a SIMD `*= scale` with a broadcast scalar produces the same per-cell f64 as a scalar `*= scale`. Parity should hold but verify.

- `src/per_sample_caller/baq/probaln.rs:283` — **S4: backward `if k >= l_ref { 0.0 } else { ... }` is a data-dependent branch inside the hot loop**
- **Confidence:** Low
- **Mechanism:** the branch is taken at most once per row (when `end == l_ref`). Peeling that iteration would let the rest of the loop be branchless. **Parity risk:** the peeled iteration must reproduce the M/I/D update with `e_val = 0.0` in the *same association order* — easy to get wrong. Verify with the parity test.

- `src/per_sample_caller/baq/scratch.rs:17` — **S5: pre-cast `qual` to `f64` once instead of casting at every f64 use**
- **Confidence:** Low
- **Mechanism:** `qual: Vec<f32>` is cast to f64 at probaln.rs:158, 181, 269, etc. — but the cast is already hoisted out of the inner k loop (it's per-row), so the win is at most O(l_query) total, not O(l_query × band).
- **Parity:** `(q2p[i] as f32) as f64` produces the same f64 bit-pattern whether cast at store or at use — but verify with the fixtures.

- `src/per_sample_caller/baq/engine.rs:291` — **S6: qname interner**
- **Confidence:** Low
- **Mechanism:** paired-end reads share a qname, so an interner (`lasso::ThreadedRodeo`) could halve qname allocations. But concurrent interners add cross-worker traffic that likely costs more than the simple `Arc<str>` allocation. Defer until L2 lands and DHAT shows qname as a remaining hot site.

- `src/per_sample_caller/baq/engine.rs:113-115` — **S7: explicit `reserve` on `tref` / `tquery` extends**
- **Confidence:** Low
- **Mechanism:** `Vec::extend` from a non-`TrustedLen` iterator pushes one element at a time. The buffers already retain capacity across reads, so the cost is paid only on the first call (or when a longer read forces a realloc). One-line `self.tref.reserve(ref_bytes.len());` fix; verify with bench.

## 9. Round-two follow-up (commit `1c970c8`)

After the user lowered the host's `perf_event_paranoid` to 2, the sampling profile from §3 step 1 became available. It overturned two of the §5 verdicts and surfaced one new finding.

### What the profile showed (engine bench, L=150)

| % self-time | site | interpretation |
|---:|---|---|
| 60.5 | `probaln_glocal` | HMM is the hot path, as expected. |

### New finding (would have been L14)

`engine.rs:127-130` had `self.state.clear() + self.state.resize(seq.len(), 0)` (and the same for `q`). The `clear()` zeroes `len`, so the subsequent `resize` always zero-fills `seq.len()` entries from scratch — the high-water-mark capacity stays but the bytes don't. Dropping the two `clear()` calls makes `resize` a no-op once the buffer hits its peak. Parity-safe (the HMM MAP loop writes every entry of `state` and `q` on its way out).

### L3 promoted from "deferred" to "applied"

The `_int_malloc` 8.9% number is the strongest motivation in the whole profile. The two `cigar.clone()` / `seq.clone()` per read in `mapped_to_prepared` were exactly the allocations we couldn't shrink without an API change. Changed `BaqEngine::process` to take `MappedRead` by value; `BaqStream::refill_batch` switched from `par_iter()` to `par_drain(..)` to consume reads by value while keeping order and the chunk-buffer's capacity. The bench harness now builds a fresh `Vec<MappedRead>` per criterion batch and moves it into `process_all` (otherwise a borrow-based bench harness would re-introduce the clone on top of the engine API change).

The `/500` engine regression is suspicious: the bench harness changed structurally to consume `MappedRead` by value (matching production), so part of the apparent regression is the new iter-batched setup cost which criterion divides across iters differently than before. Manual sanity check on `/150` (-12%) and the stream wins is what gives confidence the L3+L14 work paid off.
