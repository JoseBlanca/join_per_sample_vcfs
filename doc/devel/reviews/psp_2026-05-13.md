## 2. Verdict

**Request-changes.**

Three Blocker findings — two allocation-DoS sites and one silent integer overflow in the reader primitives that already exist — block merge for any branch that exposes the reader to untrusted file bytes. They are cheap to fix and convergent across `errors`, `extras`, and `reliability`.

The Major findings are dominated by two themes the team should treat as a pair:

1. **Wire-layout durability** — the trailer is pinned by a byte-pattern test (`trailer.rs::wire_layout_is_stable`); the block header, the manifest entry, the index entry, the header framing, and the kebab-case TOML key set are **not**. A field-reorder or rename refactor today would silently rewrite the file format and pass every existing test.
2. **Registry / dispatch coupling** — `V1_0_COLUMNS` is the source of truth for 12 column tags, but `writer.rs::encode_column_into` re-encodes the dispatch as a `match` on `u16` literals with a runtime fall-through; `verify_encoded_size` uses a `_ => true` catch-all; the `MAX_ALLELE_SEQ_LEN` cap is enforced only on the write side; `UnsupportedFormatVersion` is declared but never raised; the build-side header validator path is untested (only the read side is). Each of these is a "the compiler won't catch you" trap.

The test suite is solid for example coverage (114 passing) but has no property-based / fuzz coverage anywhere — exactly the layer where format-level off-by-one and overflow bugs hide.

## 4. Open questions and assumptions

1. **Is the writer ever expected to emit a format-version other than `READER_FORMAT_VERSION = (1, 0)`?** (Affects **M14**, **M19**, **B-tier risk model**.) If no, `validate_writer_header` should reject anything else and the reader should refuse `format_version.0 > 1`. If yes (e.g. a separate v1.1 writer slice is planned in the same crate), the writer needs a version-aware column dispatch — out of scope for this slice but flagged.
2. **What is the threat model for an untrusted `.psp` opener?** (Affects all three Blockers and **M2–M4**.) The architecture spec positions Stage 4/5 as readers of files produced by Stage 1 *within the same pipeline*; whether a `.psp` may also be accepted from a third party (a collaborator dropping a sample) is not stated. The fixes are cheap regardless, but the threat model determines whether the Blocker label is correct or whether these should be downgraded to Major. I am assuming "yes, can be third-party" because the format is documented as the durable cohort-recall artefact.
3. **Are the four sub-modules (`varint`, `trailer`, `index`, `block`) intended to be part of the crate's semver-stable public surface?** (Affects **Mi5**.) Today they are `pub mod`, making `psp::varint::decode_u64_leb128` a public path. If the answer is "no" the visibility narrowing in **Mi5** is a quick win.
4. **Is the `1.95` pin in `rust-toolchain.toml` a hard floor or a benchmark-reproducibility pin?** (Affects **Mi11**, MSRV finding.) The file comment in `rust-toolchain.toml` suggests benchmark-stability. If so, `rust-version` in `Cargo.toml` can be lower (the only post-1.85 PSP-specific feature I see is `let-else`, stable since 1.65).

## 5. Top 3 priorities

1. **B1, B2, B3 — untrusted-allocation DoS in `decode_list_column`, `decode_block_header`, `decode_index`, and silent overflow in `decode_bytes_split`.** Fix by clamping pre-allocation capacities against `bytes_remaining / min_entry_bytes` and replacing `lengths.iter().sum()` with a `checked_add` fold that also enforces `MAX_ALLELE_SEQ_LEN`. Each fix is ~5 lines of code plus one malformed-input test.

2. **M1 + M7 + M8 — wire-layout durability gap: the binary body (block header, manifest, index entry, header framing, TOML key set) has no byte-pattern pinning tests.** Mirror `trailer.rs::wire_layout_is_stable` for each fixed-layout structure. The fix is ~6 small `assert_eq!`-on-bytes tests, each ~15 lines. A single field-reorder refactor today would silently rewrite every existing `.psp`.

3. **M5 + M6 + M9 — `verify_encoded_size` debug-only + `encode_column_into` tag-literal dispatch + missing `UnsupportedFormatVersion` raise.** All three are "compiler can't catch you" against the registry / version contract. Promote `verify_encoded_size` to a real check (it costs 12 length comparisons per block); replace the `match def.tag` dispatch with an exhaustive `ColumnKey` enum or a fn-pointer on `ColumnDef`; raise `UnsupportedFormatVersion` in `parsed_from_wire` when `format_version.0 > READER_FORMAT_VERSION.0`.

## 6. Findings

### Blocker

#### B1: [block.rs:319](../../src/per_sample_caller/psp/block.rs#L319) — Untrusted varint drives unbounded `Vec::with_capacity` in `decode_list_column`
**Categories:** errors, extras (convergent)
**Confidence:** High
**Problem:** `decode_list_column` reads a per-entry `varint` count `k_u64` from the column payload and allocates `Vec::with_capacity(k)` (line 319) **before** consuming a single element from the buffer. `k_u64` is bounded only by LEB128's 10-byte cap (i.e. up to `u64::MAX`). A 30-byte hostile payload (one `0xFF…0x01` varint) can drive `Vec::<T>::with_capacity(u64::MAX as usize)` — the allocator either aborts or panics depending on platform; either way it is a non-typed failure.
**Why it matters:** The architecture spec positions `.psp` as the durable cohort-recall artefact. A reader that aborts on a one-byte tamper is a DoS / availability hole, and "errors must not pass silently" (CLAUDE.md design principle) forbids the failure mode.
**Suggested fix:**
```rust
let k = k_u64 as usize;
let remaining = bytes.len() - cursor;
// Each element occupies at least T::FIXED_BYTE_WIDTH bytes on the wire.
if k > remaining / T::FIXED_BYTE_WIDTH.max(1) {
    return Err(PspReadError::ColumnElementDecode {
        column: column_name.to_string(),
        entry,
        source: ScalarDecodeError::Truncated,
    });
}
let mut list = Vec::with_capacity(k);
```
Add a `decode_list_column_rejects_giant_inner_count_without_oom` regression test (sketch in §8).

#### B2: [block.rs:373](../../src/per_sample_caller/psp/block.rs#L373) — `decode_bytes_split` sums untrusted `u64` lengths and wraps silently in release builds
**Categories:** errors, extras (convergent); reliability surfaced at Minor
**Confidence:** High
**Problem:** `let total: u64 = lengths.iter().sum();` — with `[profile.release].overflow-checks = false` (Rust default), `u64::sum` wraps. Hostile `lengths = [u64::MAX - 5, 10]` wraps `total` to `4`, both `total > buf_len` and `total < buf_len` pass against a 4-byte buffer, then the loop hits `bytes[cursor..cursor + n]` where `n = (u64::MAX - 5) as usize` and panics. The spec also caps each entry at `MAX_ALLELE_SEQ_LEN = 10_000` ([registry.rs:148](../../src/per_sample_caller/psp/registry.rs#L148)) — the cap is enforced on the *write* side only, not at decode.
**Why it matters:** Same shape as B1: typed-error → panic-on-untrusted-bytes. The cap-asymmetry compounds the issue: the spec's hard cap is documented in the `MAX_ALLELE_SEQ_LEN` docstring as "Enforced at write time and at read time" but only the write half is in code.
**Suggested fix:**
```rust
let mut total: u64 = 0;
for (i, &len) in lengths.iter().enumerate() {
    if len > MAX_ALLELE_SEQ_LEN {
        return Err(PspReadError::ColumnElementDecode {
            column: column_name.to_string(),
            entry: i,
            source: ScalarDecodeError::Truncated, // or a new BadLength variant
        });
    }
    total = total.checked_add(len).ok_or_else(|| PspReadError::ColumnTruncated {
        column: column_name.to_string(),
        decoded: i,
        expected: lengths.len(),
    })?;
}
```
Add `decode_bytes_split_rejects_overflowing_length_sum` and `decode_bytes_split_rejects_oversized_entry` tests.

#### B3: [index.rs:88](../../src/per_sample_caller/psp/index.rs#L88) and [block.rs:529](../../src/per_sample_caller/psp/block.rs#L529), [block.rs:543](../../src/per_sample_caller/psp/block.rs#L543) — `decode_index` and `decode_block_header` allocate from untrusted counts before bounds-checking
**Categories:** reliability, errors, extras (triple convergent)
**Confidence:** High
**Problem:** Three matching patterns, one threat model:
- `decode_index`: `Vec::with_capacity(expected_n_blocks as usize)` from the trailer's `u64 n_blocks`. The minimum entry is 12 bytes (4 single-byte varints + 8-byte LE offset), so any `n_blocks > bytes.len() / 12` is guaranteed corruption — but the allocation fires first. The header parser already guards against the symmetric attack (`MAX_HEADER_BODY_BYTES` check at [header.rs:376-382](../../src/per_sample_caller/psp/header.rs#L376-L382)).
- `decode_block_header` line 529: `Vec::with_capacity(k as usize)` for `active_chain_slots` (k is `u32` → ~8 GiB for `Vec<u16>` if maxed).
- `decode_block_header` line 543: `Vec::with_capacity(n_columns as usize)` for the manifest (~64 GiB for `Vec<ColumnManifestEntry>` at `u32::MAX`).
**Why it matters:** Same DoS vector as B1, three more shapes. Together with B1/B2 these are every attacker-controlled allocation site in the partially-landed reader primitives.
**Suggested fix:** Clamp each `with_capacity` against a buffer-derived bound before allocating:
```rust
// index.rs
const MIN_ENTRY_BYTES: u64 = 12;
let max_plausible = (bytes.len() as u64) / MIN_ENTRY_BYTES;
if expected_n_blocks > max_plausible {
    return Err(PspReadError::IndexEntryCountMismatch {
        got: max_plausible as usize,
        expected: expected_n_blocks,
    });
}

// block.rs:529 (slots — each 2 bytes)
let cap = (k as usize).min(bytes.len().saturating_sub(*cursor) / 2 + 1);
let mut active_chain_slots = Vec::with_capacity(cap);

// block.rs:543 (manifest — each ≥3 bytes)
let cap = (n_columns as usize).min(bytes.len().saturating_sub(*cursor) / 3 + 1);
let mut manifest = Vec::with_capacity(cap);
```
Tests `decode_index_rejects_giant_expected_count_without_oom`, `decode_block_header_rejects_giant_active_slots_count_without_oom`, `decode_block_header_rejects_giant_n_columns_without_oom` (sketches in §8).

#### M1: [block.rs:471](../../src/per_sample_caller/psp/block.rs#L471) — `BlockHeader` has no byte-pattern wire-layout test
**Categories:** refactor_safety, extras (convergent)
**Confidence:** High
**Problem:** `encode_block_header` writes fields in struct-declaration order (lines 501–516). The round-trip test `block_header_round_trip` (line 1074) would pass *for any consistent ordering* — it only checks `encode → decode → equal`, which holds even if both sides were silently swapped. Compare `trailer.rs::wire_layout_is_stable` (lines 138–152) which pins specific byte offsets.
**Why it matters:** A future refactor that swaps `n_records` and `n_total_alleles` "for readability" rewrites every `.psp` block header on disk; the test suite stays green. The trailer test exists precisely to prevent this class of regression; the same protection is missing for the block header (the larger, riskier structure).
**Suggested fix:** Add a `block_header_wire_layout_is_stable` test that pins exact bytes for a known header. Use values whose LEB128 byte sequences are distinguishable so a field swap reorders the output detectably:
```rust
#[test]
fn block_header_wire_layout_is_stable() {
    let header = BlockHeader {
        chrom_id: 5,
        first_pos: 100,
        n_records: 1,
        n_total_alleles: 1,
        active_chain_slots: vec![7],
        manifest: vec![ColumnManifestEntry {
            tag: 0x01,
            compressed_len: 8,
            uncompressed_len: 16,
        }],
    };
    let mut buf = Vec::new();
    encode_block_header(&header, &mut buf).unwrap();
    assert_eq!(buf, [
        0x05, 0x64, 0x01, 0x01,         // chrom_id, first_pos, n_records, n_total_alleles
        0x01, 0x07, 0x00,               // slot count + slot 7 LE u16
        0x01, 0x01, 0x08, 0x10,         // n_columns + manifest[0] (tag, compressed, uncompressed)
    ]);
}
```

#### M3: [header.rs:197](../../src/per_sample_caller/psp/header.rs#L197) — Kebab-case TOML keys from `#[serde(rename = ...)]` are not asserted as literals
**Categories:** refactor_safety
**Confidence:** High
**Problem:** `WireHeader`, `WireWriter`, `WireColumn` carry serde renames (`"format-version"`, `"chromosome"`, `"column"`, `"input-crams"`, `"input-fasta"`, `"element-type"`, `"length-column"`). Every existing round-trip test reads and writes through the *same* serde definition; dropping every rename annotation tomorrow would still leave all tests green because both halves would agree on the (now wrong) snake_case key.
**Why it matters:** A field rename + dropped serde annotation silently produces files older readers cannot parse and newer readers cannot recognise as the same field. No test in the crate breaks. **Suggested fix:** Add a `wire_keys_are_pinned` test that asserts the serialised TOML body contains the kebab-case forms and does **not** contain the snake_case forms:

fn wire_keys_are_pinned() {

                "element-type =", "length-column ="] {

#### M4: [writer.rs:742](../../src/per_sample_caller/psp/writer.rs#L742) — `encode_column_into` tag-literal dispatch duplicates `V1_0_COLUMNS`; new columns silently miss the encoder
**Categories:** refactor_safety, smells (convergent); idiomatic and errors raise it as Minor cross-references
**Confidence:** High
**Problem:** `encode_column_into` matches on `def.tag: u16` with 12 hex literals (`0x01`–`0x22`) plus an `unknown => Err(PspWriteError::InvalidRecord { record_index: 0, ... })` fall-through. The dispatch is the only place the runtime binds "registry entry" to "accumulator field"; the registry source of truth (`V1_0_COLUMNS`) lives 200+ lines away in `registry.rs`. Adding a new `ColumnDef` row compiles cleanly and fails only at runtime when the first block flushes — surfacing as a misleading `InvalidRecord { record_index: 0, ... }` (the `0` is a placeholder; the error is mis-classified as a record-level failure when it is a writer-internal dispatch gap).
**Why it matters:** Binary-format-authority risk: a routine registry extension lands without compiler help to flag the matching encoder addition. The runtime failure mode is misleading enough to slow diagnosis.
**Suggested fix:** Two options, in preference order:
1. **Method on `ColumnDef`** — add `pub encode: fn(&BlockAccumulator, &mut Vec<u8>)` to `ColumnDef`. The compiler refuses to compile `V1_0_COLUMNS` until every row has an encoder. Eliminates the runtime fall-through.
2. **Exhaustive `ColumnKey` enum** — lift the dispatch key into a 12-variant enum on `ColumnDef`; the encoder matches on `ColumnKey` and is exhaustive by definition.

If neither is acceptable in this slice, at minimum add a test that iterates `V1_0_COLUMNS` and exercises `encode_column_into` for every tag on a small fixture `BlockAccumulator`, asserting each returns `Ok(())`.

#### M5: [writer.rs:795](../../src/per_sample_caller/psp/writer.rs#L795) and [writer.rs:799](../../src/per_sample_caller/psp/writer.rs#L799) — `verify_encoded_size` is `debug_assert!`-only and has a `_ => true` fall-through
**Categories:** errors, refactor_safety (convergent)
**Confidence:** High
**Problem:** Two compounding issues at one site.
1. `debug_assert!(verify_encoded_size(def, block, out));` — the writer's column-size self-check evaporates in release builds. A writer bug that produces a column whose `uncompressed_len` disagrees with the schema's prediction passes write-time validation and surfaces (much later) at read time via `UncompressedLenSchemaMismatch`. The point of "errors must not pass silently" is to fail at the producer.
2. `verify_encoded_size` itself uses `_ => true` as its last match arm. The wildcard silently approves `(_, Shape::Scalar, Some(ElementType::Svarint))` (Svarint is a registered ElementType, never enumerated in the match), any future ElementType variant, and the malformed `(Scalar, None)` triple. Two `.fixed_byte_width().unwrap()` calls inside the function are gated by `.is_some()` (fine) but invisible at the call site.
**Why it matters:** A writer self-check that runs only in debug is no check in production. The catch-all amplifies the gap: even the debug check approves invalid configurations.
**Suggested fix:**
```rust
if !verify_encoded_size(def, block, out) {
    return Err(PspWriteError::ColumnSizeSelfCheck {
        column: def.name,
        got: out.len(),
        expected: predict_uncompressed_len(def, block).unwrap_or(usize::MAX),
    });
}
```
Rewrite `verify_encoded_size` to enumerate exhaustively on the closed enums (no `_` arm):
```rust
match def.shape {
    Shape::Bytes => encoded.len() as u64 == block.allele_seq_len.iter().sum::<u64>(),
    Shape::List => true,
    Shape::Scalar => {
        let et = def.element_type.expect("non-Bytes requires element-type");
        let count = match def.cardinality {
            Cardinality::PerRecord => n_records,
            Cardinality::PerAllele => n_total_alleles,
        };
        match et {
            ElementType::Varint | ElementType::Svarint => true,
            ElementType::U8 | ElementType::U16 | ElementType::U32 | ElementType::U64
            | ElementType::I32 | ElementType::I64
            | ElementType::F32 | ElementType::F64 | ElementType::Bool => {
                encoded.len() == count * et.fixed_byte_width().expect("fixed-width path")
            }
        }
    }
}
```

#### M6: [header.rs:88-97](../../src/per_sample_caller/psp/header.rs#L88-L97) and [header.rs:478](../../src/per_sample_caller/psp/header.rs#L478) — Writer accepts any `format_version`; reader never raises `UnsupportedFormatVersion`
**Categories:** defaults, extras (convergent)
**Confidence:** High
**Problem:** Two halves of the same gap, sharing the unused constant `READER_FORMAT_VERSION = (1, 0)` at [header.rs:78](../../src/per_sample_caller/psp/header.rs#L78):
- **Writer side:** `WriterHeader::format_version: (u16, u16)` is caller-supplied; `validate_writer_header` rejects only `(0, 0)`. A caller passing `(2, 0)` produces a file whose binary body is v1.0-shaped but whose header advertises v2.0.
- **Reader side:** `PspReadError::UnsupportedFormatVersion` ([errors.rs:217-224](../../src/per_sample_caller/psp/errors.rs#L217-L224)) exists with the right shape but is never constructed. A `.psp` declaring `format-version = "2.0"` parses cleanly, succeeds the schema cross-check against the v1.0 registry, and the caller gets a `ParsedHeader` claiming `(2, 0)` without warning.
**Why it matters:** Spec §"Header-binary consistency" check #5 and §"Versioning policy" both require the reader to refuse higher-major files. Without the check a future v2.0 writer can land bytes the v1.0 reader silently misinterprets. The architecture spec is explicit that this is the version boundary the format-evolution policy exists to defend.
**Suggested fix:**
1. In `validate_writer_header`:
   ```rust
   if header.format_version != READER_FORMAT_VERSION {
       return Err(bad("format-version", format!(
           "writer only emits {}.{}; got {}.{}",
           READER_FORMAT_VERSION.0, READER_FORMAT_VERSION.1,
           header.format_version.0, header.format_version.1)));
   }
   ```
2. In `parsed_from_wire` after `parse_format_version`:
   ```rust
   if format_version.0 > READER_FORMAT_VERSION.0 {
       return Err(PspReadError::UnsupportedFormatVersion {
           file_major: format_version.0, file_minor: format_version.1,
           reader_major: READER_FORMAT_VERSION.0, reader_minor: READER_FORMAT_VERSION.1,
       });
   }
   ```
3. Test pinning each variant for a `(2, 0)` input on both paths.

#### M7: [errors.rs:69](../../src/per_sample_caller/psp/errors.rs#L69), [errors.rs:306](../../src/per_sample_caller/psp/errors.rs#L306), [errors.rs:25](../../src/per_sample_caller/psp/errors.rs#L25), [errors.rs:43](../../src/per_sample_caller/psp/errors.rs#L43) — Public error enums lack `#[non_exhaustive]`

[trailer.rs:138-152](../../src/per_sample_caller/psp/trailer.rs#L138-L152) (`wire_layout_is_stable`) pins specific byte offsets to specific magic values; a field-reorder refactor would fail this test loudly. - The writer's `validate_record` runs the allele byte check via a lookup-table walk ([writer.rs:399](../../src/per_sample_caller/psp/writer.rs#L399)) deliberately so LLVM can autovectorise on long alleles, with the cold path scanning a second time only to locate the offending byte for the error message. - The errors module's "no `Other(String)` catch-alls" stance ([errors.rs:11-13](../../src/per_sample_caller/psp/errors.rs#L11-L13)) is the right baseline. M10 and Mi10 are about *typing the `reason: String` fields further* — the file already refuses the worst form of error punting.
