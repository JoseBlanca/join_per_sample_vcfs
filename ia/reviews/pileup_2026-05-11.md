## 2. Verdict

**Request-changes.**

- The two `clippy::needless_borrow` lints at `walker.rs:290, 306` break `cargo clippy ... -D warnings` and currently block CI. This is mechanical to fix.
- Four call sites use wildcard `_ =>` arms on the internal `ReadEvent` enum, which will silently misroute a future variant into the indel path; one of the four (`column_depth_cap`) is load-bearing for per-column truncation behaviour.
- Five of seven `WalkerError` variants have zero test coverage; the `ChannelClosed`, `ZeroRefSpan`, `RecordTooWide`, and `Fasta` paths cannot regress without somebody noticing.
- The `Internal` and `ChannelClosed` error variants violate the module's own stated design principle (errors carry locus context) by stuffing data into a stringified `detail` / `context` field.
- `WalkerConfig`'s defaults are still bare literals and the resolved config is not emitted at run start, so operators triaging a `column_depth_truncations > 0` run cannot answer "what cap was this run with?" from logs.

No Blockers. The walker's correctness story is otherwise strong — 88/88 tests pass, the lazy-cursor parity oracle is excellent, the panic-free invariants are documented at almost every site, and several prior-review findings are visibly addressed in the code (chain-slot orphan release, walker-pos overflow guard, hoisted `contributors_buf`/`allele_seq_buf`).

#### <a id="m2"></a>M2: src/per_sample_caller/pileup/walker.rs:248, 633, 667, 679 — Wildcard `_ =>` on internal `ReadEvent` silently absorbs future variants

- **Confidence:** High
- **Categories:** refactor_safety (raised 4 ×), idiomatic (raised as single Minor).
- **Problem:** Four call sites match on the internal three-variant `ReadEvent` with a `_ =>` arm:
  - `walker.rs:248-254` (`bq_at_walker` extraction): non-Match → `None` → `unwrap_or(0)`.
  - `walker.rs:633-638` (`match_base_at_pos`): non-Match → `None`, paired with `.expect("match-only overlap…")` at `:510-512`.
  - `walker.rs:667-674` (`pair_has_indel`): `!matches!(e, ReadEvent::Match { .. })` classifies *any* non-Match variant as indel and routes the pair to the "drop the loser" path.
  - `walker.rs:679-690` (`column_depth_cap`): same shape; flips the column to the tighter `max_indel_column_depth` cap.

  A future `ReadEvent` variant (e.g. `ReferenceN`, `Mismatch`, split-read evidence) would be silently classified as indel-bearing on the last two, default `bq_at_walker` to 0 on the first, and panic on the `.expect()` on the third — none of which the compiler would surface.
- **Why it matters:** `column_depth_cap` is the load-bearing per-column truncation; a silent re-classification driven by an unrelated variant addition would change which columns get truncated without any signal. This is exactly the refactor-safety hazard the project's lint policy (`clippy::wildcard_enum_match_arm = "deny"` if/when enabled) targets.
- **Suggested fix:** Enumerate indel variants positively at every site. Concrete diff for one:
  ```rust
  // walker.rs:679
  fn column_depth_cap(contributors: &[ReadContribution], config: &WalkerConfig) -> usize {
      let any_indel = contributors.iter().any(|c| {
          c.events_at_pos.iter().any(|e| matches!(
              e,
              ReadEvent::Insertion { .. } | ReadEvent::Deletion { .. },
          ))
      });
      if any_indel {
          config.max_indel_column_depth as usize
      } else {
          config.max_snp_column_depth as usize
      }
  }
  ```
  Same shape at the other three sites. Open-record's `ln_bq_for_read` (`open_record.rs:791-795`) already enumerates exhaustively — model the others after it.

#### <a id="m5"></a>M5: src/per_sample_caller/pileup/errors.rs:58 — `WalkerError::Internal { detail: String }` is the catch-all anti-pattern

- **Confidence:** High

#### <a id="m6"></a>M6: src/per_sample_caller/pileup/errors.rs:68 — `WalkerError::ChannelClosed { context: String }` is a stringified discriminator missing locus context

- **Confidence:** High
- **Categories:** errors (Major), unsafe_concurrency (Minor — same site).
- **Problem:** The two call sites at [walker.rs:324](../src/per_sample_caller/pileup/walker.rs#L324) and [walker.rs:389](../src/per_sample_caller/pileup/walker.rs#L389) hand-craft the strings `"walker output channel closed mid-run"` and `"… during flush"` — a string-encoded enum of two phases. - **`merge_slot_counters` name.** Verb-direction ambiguous; consider `with_slot_counters_from` or similar.
- **`zero_event_bq` name.** Function name promises generic "zero an event's BQ" but body is narrow ("Match-only"). Rename to `zero_match_event_bq` to mirror sibling `set_match_event_bq` and avoid the wider promise.
- **`new` as a binding identifier.** Shadowing of `new` in destructures (`let (new, expired) = …`) at `walker.rs:321, :387` and 10+ test sites — stylistic loudness, no semantic issue. Single rename pass.
- **`WalkerError::Internal { qname: String }` storage type.** Most call sites construct via `read.qname.to_string()` — fine for `Display`, but every other field elsewhere in the module holds `Arc<str>`. Either keep `Arc<str>` in the variant or leave a comment noting the `to_string` is intentional for `thiserror`'s render.
- **`contributors_buf` capacity hint.** Hoisted at `walker.rs:172` but never pre-sized. A `Vec::with_capacity(64)` initial hint would skip the first-step grow ladder.
- **`AHashMap<SlotId, Vec<usize>>` capacity hint at `walker.rs:455`.** Typical column has ≤ a handful of shared slots; `with_capacity(4)` skips the first grow. Trivial.
- **`walker.rs:344` debug_assert wording.** Asserts `walker_pos > 0` but the comment says "never decreases below 1" — wording mismatch.
- **`subtract_contribution` `q_sum` `f64` raw vs. saturating asymmetry.** Comment justifies u32 saturate; doesn't address why `q_sum -= c.q_sum` is straight `f64` subtract. (Correct — `f64` is signed by design.) One-line note would resolve the asymmetry.
- **`active_count()` O(MAX_ACTIVE_SLOTS=4096) scan on every `set_refcount` call.** Maintain an explicit counter; the bookkeeping is one increment/decrement per refcount transition. Cross-category perf note; punt unless a bench shows it matters.
- **`free` Vec sorted-descending via `binary_search_by` + `insert`.** O(n) per release at MAX_ACTIVE_SLOTS = 4096. Acceptable today; a binary heap with flipped comparator would be cleaner if it ever shows up in profiles.
