## 4. Open Questions and Assumptions

1. **Does the Stage 2 encoder rely on every `new_chains[S]` having a matching `expired_chains[S]` within the same chromosome?** The spec says "chains do not span chromosomes" and the Stage 2 layout is described in `per_sample_pileup_format.md`, but the consumer-side contract for unmatched lifecycle marks at chromosome-end is not pinned. **Affects M1.** If consumers reset state per chromosome regardless, M1 is downgraded to Minor; if they require well-formed pairs, M1 is Blocker.
2. **Is the upstream `cram_input` layer a hard guarantor of `(chrom_id, alignment_start)` monotonicity, or only of within-chromosome monotonicity?** **Affects M2.** If `cram_input` cannot regress `chrom_id`, M2 is Minor (defensive); otherwise Major.
3. **The `column_depth_cap` truncates `contributors[..cap]` and assumes admission order is allele-uncorrelated.** The justification in [mod.rs:74-77](../src/per_sample_caller/pileup/mod.rs#L74-L77) is plausible but unproven for adversarial inputs (e.g., a sample with strongly batched coverage from sorted region-restricted CRAMs). **Affects Mi5.** Is a cheap deterministic random sample (e.g., reservoir over `read_id`) acceptable as a follow-up?

## 5. Top 3 Priorities

1. **M1** â€” Fix the orphan-first-mate slot leak in `flush_chromosome`. Slots whose first mate's partner never arrives within the chromosome are silently consumed: their `new_chains` mark fires, but no `expired_chains` mark is ever emitted because `release_slot` only decrements refcount from 2 â†’ 1. See [Finding M1](#m1-walkerrsl334l362--orphan-first-mate-leaks-unmatched-new_chains-mark-at-chromosome-flush).
2. **M2** â€” Add a chromosome-id regression check at admit time. The current `OutOfOrder` test compares `(chrom_id, alignment_start)` tuples but `set_chrom_if_needed` resets `last_admitted_locus` to `None` whenever the chrom changes, so a regression like `chrom_id 1 â†’ 0` flushes-and-restarts silently rather than erroring. See [Finding M2](#m2-walkerrsl42l67--chromosome-id-regression-silently-bypasses-the-out-of-order-check).
3. **Mi3** â€” Decide-and-document `bq_zero_in_window`'s scope: it currently zeros every indel `bq_proxy` in a contributor's haplotype window after a match-only mate-overlap loss, even at positions where no overlap occurred. This is consistent with what the eager design produced (per-haplotype min collapses to 0 anyway), but the spec wording says "zero the loser **locally for that position**." Either tighten the implementation to the per-position reading or amend the spec.

## 6. Findings

### Major

#### M1: [walker.rs:334-362](../src/per_sample_caller/pileup/walker.rs#L334-L362) + [slot_allocator.rs:204-220](../src/per_sample_caller/pileup/slot_allocator.rs#L204-L220) â€” Orphan first mate leaks unmatched `new_chains` mark at chromosome flush

- **Confidence:** High
- **Assumptions:** The downstream consumer expects new/expired chain marks to balance within a chromosome. This is consistent with the Stage 5 consumer model implied by `phase_chain.md` and the `per_sample_pileup_format.md` block layout. Confirmation pending Open Question 1.
- **Problem:**
  When a paired read's mate never arrives in the same chromosome as its first mate, the slot allocator pre-bumps refcount to 2 ([slot_allocator.rs:167](../src/per_sample_caller/pileup/slot_allocator.rs#L167)) on first-mate admission. The `pending_mates` entry that would normally drive `evict_stale_pending` is only inspected during `allocate_for_read`, which is **not called** during `flush_chromosome`. So at flush:
  1. `drain_aged` / `flush_all` pop every active read.
  2. `flush_all` calls `release_slot` once per active read.
  3. For the orphan first mate, refcount goes 2 â†’ 1, **does not** reach 0, **does not** push to `expired_marks`.
  4. `slots.reset()` then zeroes `slot_refcount` and clears `pending_mates`, erasing the trail.

  The resulting record stream contains a `new_chains: [S]` somewhere earlier in the chromosome, but no `expired_chains: [S]` anywhere in the chromosome. After `reset`, the next chromosome may reissue the same `S`.

  The existing test `flush_all_releases_every_slot` ([active_set.rs:295](../src/per_sample_caller/pileup/active_set.rs#L295)) only exercises **solo** reads (`has_mate = false`), so refcount = 1 â†’ 0 fires expired correctly. The paired-orphan path is uncovered.

- **Why it matters:**
  Phase chains are a primary Stage 1 â†’ Stage 5 hand-off. A consumer that bookkeeps "alive set = sum(new) âˆ’ sum(expired)" within a chromosome will end the chromosome believing chain `S` is still alive. If the next chromosome reuses `S`, the alive set is corrupted. Even if the consumer resets state per-chromosome (Open Question 1), the contract is still asymmetric with the spec, which says ([Â§"Defensive bounds"](../ia/specs/pileup_walker.md)) the eviction path "stays at refcount 1, the slot will release when the first mate's `ActiveRead` exits" â€” implying expired *will* fire. At chromosome end, that expectation breaks.

  This also potentially breaks chromosomes that contain *only* an unmated first mate (extreme but valid input): the run produces records with `new_chains: [S]` and never any `expired_chains` for that `S`.

- **Suggested fix:**
  At the start of `flush_chromosome`, evict every `pending_mates` entry. The cleanest place is in `SlotAllocator` itself â€” add a `flush_pending_mates(&mut self)` method that decrements refcount on every pending entry, mirroring the per-entry logic of `evict_stale_pending`:

  ```rust
  // slot_allocator.rs
  pub fn flush_pending_mates(&mut self) {
      for entry in self.pending_mates.values() {
          let idx = entry.chain_slot_id as usize;
          if idx < self.slot_refcount.len() && self.slot_refcount[idx] >= 2 {
              self.slot_refcount[idx] -= 1;
              self.counters.mate_lookup_evictions += 1;
          }
      }
      self.pending_mates.clear();
  }
  ```

  Then in `walker.rs::flush_chromosome`, call it before `flush_all`:

  ```rust
  // walker.rs
  fn flush_chromosome(&mut self, tx: &SyncSender<PileupRecord>) -> Result<(), WalkerError> {
      let remaining = self.open.drain_all();
      let mut records: Vec<PileupRecord> = remaining.into_iter().map(|r| r.finalise()).collect();
      let (new, expired) = self.slots.drain_lifecycle_marks();
      // Treat any still-pending first mate as solo before draining the active set.
      // Without this, the orphan's release_slot decrements refcount 2 â†’ 1 (not 0),
      // so its expired_chains mark is never emitted.
      self.slots.flush_pending_mates();
      self.active.flush_all(&mut self.slots)?;
      let (new2, expired2) = self.slots.drain_lifecycle_marks();
      // ... unchanged ...
  }
  ```

  **New regression test** (in `slot_allocator::tests`):

  ```rust
  #[test]
  fn flush_pending_mates_emits_expired_for_orphan_first_mate() {
      let mut a = SlotAllocator::new();
      let m1 = make_read("orphan", true, 100);
      let (slot, _) = a.allocate_for_read(&m1).unwrap();
      let (new, _) = a.drain_lifecycle_marks();
      assert_eq!(new, vec![slot]);

      // Simulate flush_chromosome's new step + the active-set release.
      a.flush_pending_mates();
      a.release_slot(slot).unwrap();

      let (_, expired) = a.drain_lifecycle_marks();
      assert_eq!(
          expired,
          vec![slot],
          "orphan first mate's slot must surface as expired at chromosome flush",
      );
  }
  ```

  **End-to-end regression test** (in `tests.rs`):

  ```rust
  #[test]
  fn orphan_first_mate_emits_balanced_lifecycle_marks() {
      let fa = MockFasta::new("ACG");
      let mut r = snp_read("orphan", 1, b"ACG", &[30; 3]);
      r.has_mate = true;
      r.is_first_mate = true;

      assert_eq!(total_expired, 1, "orphan first mate must produce a matching expired mark");
  }
  ```

#### M2: [walker.rs:42-67](../src/per_sample_caller/pileup/walker.rs#L42-L67), [walker.rs:163-173](../src/per_sample_caller/pileup/walker.rs#L163-L173), [walker.rs:175-201](../src/per_sample_caller/pileup/walker.rs#L175-L201) â€” Chromosome-id regression silently bypasses the out-of-order check

- **Confidence:** High
- **Assumptions:** The walker is documented to validate the per-spec invariant "Monotonically non-decreasing `(chrom_id, alignment_start)`" ([pileup_walker.md Â§"Input interface"](../ia/specs/pileup_walker.md)). Confirmation of upstream guarantees pending Open Question 2.
- **Problem:**
  The walker's outer loop flushes on chromosome change ([walker.rs:45-50](../src/per_sample_caller/pileup/walker.rs#L45-L50)) without first checking whether the change is *forward*. After flush, `set_chrom_if_needed` ([walker.rs:163-173](../src/per_sample_caller/pileup/walker.rs#L163-L173)) resets `last_admitted_locus = None`, which causes `admit_read`'s order check ([walker.rs:178-188](../src/per_sample_caller/pileup/walker.rs#L178-L188)) to silently accept any starting position on the new chrom_id.

  Concretely, an input stream `(chrom_id=1, pos=100), (chrom_id=0, pos=50)` violates the spec's monotonicity invariant but the walker:
  1. Emits chrom 1's records.
  2. Flushes on the chrom transition.
  3. Re-anchors at chrom 0, walker_pos=1, last_admitted_locus=None.
  4. Admits the chrom 0 read without complaint.

  Compare with the within-chromosome regression test `out_of_order_input_is_a_hard_error` ([tests.rs:683-694](../src/per_sample_caller/pileup/tests.rs#L683-L694)) â€” which would catch `(chrom_id=0, pos=5), (chrom_id=0, pos=1)` but not the cross-chromosome case.

- **Why it matters:**
  This is precisely the class of failure the order check exists for: cracked or incorrectly-merged input streams. If `cram_input` ever has a bug that emits chromosomes out of header order, the walker silently produces a per-chromosome record stream that Stage 2 will assume is well-formed. Failure here is the kind that "passes silently" â€” the Stage 1 output looks plausible until much later analyses catch the mismatch (or never do). This contradicts [design_principles.md](../ia/specs/design_principles.md) principle 3 ("Errors must not pass silently").

- **Suggested fix:**
  Before flushing on chromosome change, validate that the new chrom_id is strictly greater than the last one (or that there is no last one):

  ```rust
  // walker.rs in run()
  if let Some(peek) = iter.peek()
      && let Some(prev_chrom_id) = state.last_admitted_chrom_id
      && prev_chrom_id != peek.chrom_id
  {
      if peek.chrom_id < prev_chrom_id {
          return Err(WalkerError::OutOfOrder {
              qname: peek.qname.to_string(),
              prev_chrom_id,
              prev_pos: state.walker_pos,
              chrom_id: peek.chrom_id,
              pos: peek.alignment_start,
          });
      }
      state.flush_chromosome(tx)?;
  }
  ```

  **New regression test** (in `tests.rs`):

  ```rust
  #[test]
  fn chromosome_id_regression_is_a_hard_error() {
      let fa = MockFasta::new("ACGTA");
      // chrom 1 then chrom 0 â€” an invalid stream per spec.
      let mut r1 = snp_read("a", 1, b"AC", &[30; 2]);
      r1.chrom_id = 1;
      let mut r2 = snp_read("b", 1, b"AC", &[30; 2]);
      r2.chrom_id = 0;
      let (tx, _rx) = std::sync::mpsc::sync_channel::<super::PileupRecord>(64);
      let result = run(vec![r1, r2], &fa, &tx, &WalkerConfig::default());
      assert!(result.is_err(), "chrom_id regression must fail");
      assert!(result.unwrap_err().to_string().contains("out-of-order"));
  }
  ```

#### Mi2: [slot_allocator.rs:204-213](../src/per_sample_caller/pileup/slot_allocator.rs#L204-L213) â€” `WalkerError::Internal` raised from `release_slot` carries empty/zero context fields

- **Confidence:** High
- **Problem:**
  ```rust
  return Err(WalkerError::Internal {
      detail: format!("release_slot on non-active slot {slot}"),
      qname: String::new(),
      chrom_id: 0,
      pos: 0,
  });
  ```
  The `Internal` variant is documented to carry `qname`, `chrom_id`, `pos` to "point at the offending input" ([errors.rs:1-4](../src/per_sample_caller/pileup/errors.rs#L1-L4)). #### Mi3: [open_record.rs:582-588](../src/per_sample_caller/pileup/open_record.rs#L582-L588) + [open_record.rs:495-499](../src/per_sample_caller/pileup/open_record.rs#L495-L499) â€” `bq_zero_in_window` zeros indel proxies, not just Match BQ

- **Confidence:** Medium
- **Assumptions:** The match-only mate-overlap path (S7) is the only place `bq_zero_in_window` is set. Confirmed: [walker.rs:461,471](../src/per_sample_caller/pileup/walker.rs#L461) only sets `zero_in_window=true` on the loser side of a match-only overlap, and [walker.rs:436-443](../src/per_sample_caller/pileup/walker.rs#L436-L443) routes pairs with any indel through `pair_has_indel` to the indel-removal branch instead.
- **Problem:**
  When `bq_zero_in_window` is set on a contributor's `ReadContribution`, [zero_event_bq](../src/per_sample_caller/pileup/open_record.rs#L582-L588) is applied to **every** event in the haplotype window â€” including `Insertion::bq_proxy` and `Deletion::bq_proxy`. The spec wording says "zero the loser **locally for that position**" â€” i.e., the match-only Match event at walker_pos. The implementation widens that to "every event in the wider record's window."

  In practice, an open record at the walker_pos with footprint [walker_pos, walker_pos+1) (a SNP-only record) will only contain that one Match event in the contributor's window, so the over-zeroing is silent. But once the record widens â€” say a different contributor's later deletion extends the record's footprint past walker_pos â€” the loser's window can include indel events from elsewhere in the read. Their `bq_proxy` then gets zeroed by `zero_event_bq`, which folds into `ln_bq_for_read`'s `min`. The compound-allele BQ becomes 0, and `q_sum` collapses to `max(0, mq_log_err)`.

  This is **also** what the eager design produced (the cloned full-window event vector had its Match bq_baq zeroed; the per-haplotype `min` would then be 0 regardless of the indel's preserved bq_proxy, so the eager output is the same). So this isn't a behavior change from the previous design. But the *spec* still says "for that position," and the `zero_event_bq` function happens to widen that to "every event," which is a different statement than the spec.

- **Why it matters:** Two issues:
  1. The spec and the code disagree on what `bq_zero_in_window` means.
  2. If the haplotype reduction rule (`min`) is ever revisited (e.g. switched to median or weighted), the over-zeroing would silently bias the new aggregator.

- **Suggested fix:** Either tighten `zero_event_bq` to only mutate Match events:
  ```rust
  fn zero_event_bq(ev: &mut ReadEvent) {
      if let ReadEvent::Match { bq_baq, .. } = ev {
          *bq_baq = 0;
      }
      // Indel bq_proxy is left alone â€” overlap resolution is per-position
      // for Match events. Per-haplotype `min` already collapses to 0 via
      // the zeroed Match.
  }
  ```
  And add a regression test pinning the indel BQ proxy is preserved when the loser's haplotype window includes a non-walker-pos indel.

  Or amend the spec to say "every event in the loser's haplotype window" if that is the intended generalisation.

#### Mi4: [walker.rs:397](../src/per_sample_caller/pileup/walker.rs#L397) â€” `resolve_mate_overlap_at_pos` uses `std::collections::HashMap`, not the project-standard `ahash::AHashMap`

- **Confidence:** High
- **Problem:** Every other map in this module uses `ahash::AHashMap`, but the local `by_slot` map uses the std `HashMap` with the default `RandomState` hasher. Iteration order on `by_slot.values()` is therefore non-deterministic between processes â€” it doesn't currently affect output (the per-pair logic is symmetric and `to_remove` / `bq_updates` are sorted/applied independently of insertion order), but the inconsistency is a foot-gun for any future refactor that *does* depend on iteration order.
- **Why it matters:** Determinism across runs is a design property â€” the slot allocator and active set go to lengths to be deterministic. Using a non-deterministic hasher in a hot path is a latent regression magnet.
- **Suggested fix:** Replace with `ahash::AHashMap`:
  ```rust
  let mut by_slot: ahash::AHashMap<SlotId, Vec<usize>> = ahash::AHashMap::new();
  ```
  The change is mechanical and preserves all other behavior.

#### Mi5: [walker.rs:248-264](../src/per_sample_caller/pileup/walker.rs#L248-L264) + [mod.rs:74-77](../src/per_sample_caller/pileup/mod.rs#L74-L77) â€” Per-column depth cap relies on uncorrelated admission order

- **Confidence:** Medium
- **Problem:** The cap drops contributors past index `cap`:
  ```rust
  if contributors.len() > cap {
      contributors.truncate(cap);
      self.summary.column_depth_truncations += 1;
  }
  ```
  Admission order = read arrival order = coordinate order from `cram_input`. The doc-comment justifies this as "approximately unbiased" because reads aren't allele-correlated in iteration order. That holds for typical Illumina random-fragment data, but breaks for any input where coverage is *batched* by source (per-CRAM merge that hasn't fully interleaved, lane-stratified pre-filtering, region-restricted CRAMs concatenated rather than merged). At those inputs, the first `cap` contributors at a deep column may be heavily skewed toward one lane / one library / one allele.

  This is also untested: there is no test that pins the bias direction or verifies non-bias.

- **Why it matters:** The per-column depth cap is a defensive hedge against pathological depth, not a routine pruner â€” but at very deep loci (centromeres, repeat regions where MAX_ACTIVE_SLOTS hasn't yet fired), it is the only thing keeping the column tractable. A biased truncation silently shifts allele frequencies at exactly those loci.
- **Suggested fix:** Use a cheap deterministic random sample. The contributor index can seed a hash:
  ```rust
  // Sample `cap` contributors uniformly using a deterministic hash over
  // (read_id, walker_pos). Avoids both bias and rng-machinery.
  if contributors.len() > cap {
      use std::hash::Hasher;
      contributors.sort_by_key(|c| {
          let mut h = ahash::AHasher::default();
          h.write_u32(c.read_id);
          h.write_u32(walker_pos);
          h.finish()
      });
      contributors.truncate(cap);
      self.summary.column_depth_truncations += 1;
  }
  ```
  Alternatively, accept the bias and add a doc-comment-and-test pair that pins the bias is at most "first-N of admission order" so that operators can reason about it.

#### Mi6: [open_record.rs:481-485](../src/per_sample_caller/pileup/open_record.rs#L481-L485) â€” Cloning `rec_ref_seq` per affected record per call

- **Confidence:** High
- **Problem:**
  ```rust
  for key in affected {
      let (rec_pos, rec_ref_seq) = {
          let rec = open.records.get(&key).expect("affected key must exist");
          (rec.pos, rec.ref_seq.clone())  // <-- clones the entire ref_seq
      };
      let rec_end = rec_pos + rec_ref_seq.len() as u32;
      for contrib in contributors {
          // ... uses rec_ref_seq for apply_events_to_ref ...
          let rec = open.records.get_mut(&key).expect("affected key must exist");
          // ... mutates rec ...
      }
  }
  ```
  The clone is a borrow-checker workaround: we need an immutable view of `ref_seq` to compute `allele_seq` and a mutable view of the record's `alleles` / `folded_reads` to update them. The clone cost is bounded by `MAX_RECORD_SPAN = 5000` per affected record per walker step, but it's avoidable.
- **Why it matters:** Walker is single-threaded and on the hot path. Per-step cloning of a 5 kbp `Vec<u8>` is wasteful where a borrow split would do. This is the kind of incremental cost that adds up at high-coverage / long-read scaling.
- **Suggested fix:** Restructure to pull the `ref_seq` borrow into a separate scope, then drop it before the mutable borrows:
  ```rust
  for key in affected {
      let rec_pos;
      let rec_end;
      let mut allele_updates: Vec<(u32, /*read_id*/ Vec<u8>, FiveScalars, SlotId)> = Vec::new();
      {
          let rec = open.records.get(&key).expect("affected key must exist");
          rec_pos = rec.pos;
          rec_end = rec_pos + rec.ref_seq.len() as u32;
          for contrib in contributors {
              // ... compute allele_seq using &rec.ref_seq (no clone) ...
              allele_updates.push((contrib.read_id, allele_seq, new_contribution, contrib.chain_slot_id));
          }
      } // rec borrow ends
      let rec = open.records.get_mut(&key).expect("affected key must exist");
      for (read_id, allele_seq, new_contribution, slot) in allele_updates {
          // apply
      }
  }
  ```
  Or extract the per-(record, contrib) computation into a free function that takes `&ref_seq` and returns the action to apply, then apply mutations in a second pass.

#### Mi8: [open_record.rs:182-203](../src/per_sample_caller/pileup/open_record.rs#L182-L203) â€” `find_overlapping`'s `q < event_end` check is redundant given the range bound

- **Confidence:** High
- **Problem:** The function iterates `self.records.range(lo..=event_start).rev()`, so every visited `q` satisfies `q â‰¤ event_start`. The check `q < event_end` then asks "is `q < event_end`?" â€” but for a non-empty event (`event_start < event_end`), `q â‰¤ event_start < event_end`, so the check is always true. For an empty event (`event_start == event_end`, never produced by current code paths), the check would correctly reject `q == event_start`, but no caller passes empty events.
- **Why it matters:** Dead/redundant condition complicates reasoning. The simpler `rec.footprint_end_exclusive() > event_start` predicate is sufficient and self-evident.
- **Suggested fix:**

  ```rust
      if rec.footprint_end_exclusive() > event_start {
          return Some(q);
      }
  }

  ```
  Add a `debug_assert!(event_start < event_end)` at the top to pin the empty-event invariant.

#### Mi9: [open_record.rs:329-330](../src/per_sample_caller/pileup/open_record.rs#L329-L330) â€” `apply_events_to_ref` re-sorts already-sorted events; tied-anchor order matters

- **Confidence:** Medium
- **Problem:**
  ```rust
  let mut sorted: Vec<&ReadEvent> = events.iter().collect();
  sorted.sort_by_key(|e| e.anchor_pos());
  ```
  The cursor returns events in CIGAR order, which is also ref-position order, so they're already sorted. The defensive re-sort is wasteful but cheap (Vec collect + sort_by_key is `O(n)` for already-sorted input) â€” acceptable.

  More subtly: `sort_by_key` is **stable**, so events with equal anchors keep their input order. The function's correctness relies on this for tied-anchor events: at anchor `p`, a `Match { ref_pos: p, base: 'X' }` followed by an `Insertion { anchor_ref_pos: p, ... }` produces `read_base + inserted_seq`, while the reversed order produces `ref_base + inserted_seq + read_base` (the second iteration's anchor push is suppressed by `consumed_until`, but the order in which the anchor base is emitted differs from the actual read).

  In production this is fine because the cursor always emits Match before Insertion at the same anchor. But the function's contract should make this explicit: "events at the same anchor must be in `Match â†’ Insertion â†’ Deletion` order." The current contract just says "sorted by anchor."

- **Why it matters:** A future caller that constructs events out-of-order (a unit test, a fuzz harness, an in-memory rebuild path) will silently produce wrong haplotypes. The sort is doing partial work â€” it canonicalises across anchors but not within an anchor.
- **Suggested fix:**
  Two options:
  1. **Document the precondition** and assert it in debug:
     ```rust
     // Precondition: events sorted by (anchor, kind-priority) where
     // kind-priority = Match < Insertion < Deletion. The cursor satisfies this
     // by walking ops in CIGAR order. We do not re-sort: stable sort preserves
     // input order on ties, so re-sorting buys nothing here.
     debug_assert!(events.windows(2).all(|w| {
         let a = (w[0].anchor_pos(), event_kind(w[0]));
         let b = (w[1].anchor_pos(), event_kind(w[1]));
         a <= b
     }));
     ```
  2. **Do a true total sort** (anchor + kind):
     ```rust
     sorted.sort_by_key(|e| (e.anchor_pos(), event_kind_rank(e)));
     ```

  I recommend option 1: the cursor already satisfies the stronger precondition, and adding a real total order is a trap (downstream code might rely on a specific tie order that changes silently if the kind-priority is renumbered).

## 9. What's Good

Up to five concrete, transferable patterns worth keeping.

1. **Cursor parity oracle in [cigar_cursor.rs:850-932](../src/per_sample_caller/pileup/cigar_cursor.rs#L850-L932).** Every `(mode, pattern, query_window)` triple is asserted byte-equal against the eager `decompose` reference. This is the right shape for any optimisation that replaces an existing implementation: keep the slow version as a test-only oracle. It pre-empts an entire class of regression bugs and provides obvious onboarding documentation for the cursor's contract.
2. **`pending_free` deferral with explanatory comment in [slot_allocator.rs:226-251](../src/per_sample_caller/pileup/slot_allocator.rs#L226-L251).** The `same_emission_reuse_does_not_collide_on_slot_id` test ([slot_allocator.rs:447](../src/per_sample_caller/pileup/slot_allocator.rs#L447)) plus the doc-comment block explaining *why* the pre-pending-free design was wrong is the gold standard for "fixed bug, kept the lesson." 3. **`folded_reads` per-record subtract-then-add in [open_record.rs:541-559](../src/per_sample_caller/pileup/open_record.rs#L541-L559).** Solving the "fold each (record, read) pair exactly once over the record's lifetime" problem by storing the previous contribution and subtracting it on re-fold (rather than trying to deduplicate folds at the walker level) is robust and self-correcting under widening.
