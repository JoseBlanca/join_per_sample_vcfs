## 2. Verdict

**Request changes.** Two Blockers: silent I/O-error swallowing on the ref-fetcher path (data loss masked as routine boundary skip), and absent direct tests on `probaln_glocal`'s input-validation paths. Beyond those, a dense cluster of Major findings split between (a) refactor-safety hazards that put the htslib-parity invariant at risk (CIGAR wildcards, struct-literal spread, magic-literal parity with no compile-time guard), (b) error-typing and narrowing-cast hazards on user-controlled fields, (c) test gaps on the production CIGAR walks (only `Match` and `SoftClip` are happy-path tested), and (d) a naming pass the author has already invited — see Top 3 below.

The implementation itself is careful, the parity comments are thorough, and the rayon design is sound; the issues are in the surface around the port, not in the algorithm.

## 5. Top 3 priorities

1. **B1** — `engine.rs:121` silently collapses every `RefSeqFetcher::fetch` error into `RefWindowPastChromEnd`. A genuine I/O error (permission denied, decompression, corrupt index) becomes a benign-looking skip with no log line. **Real correctness hazard.**
2. **B2 + M16 + M27** — `probaln_glocal`'s input-validation paths (length mismatch, `i32`-range, negative `cfg.bw`, `bw*2+1` overflow) have **no direct tests**. The parity fixtures cannot reach them; `BaqEngine` happens to size things consistently today, so a future refactor that removes a guard cannot fail any test. Two of the four conditions also produce silently wrong results rather than panic.
3. **M1 + M2 + naming pass** — The user explicitly requested renaming recommendations. The single most impactful one is `BaqOverflow` (the enum name) which actively misleads on its `InvalidInput` variant. The lumping of three distinct conditions into `InvalidInput` (M2) and the convergent narrowing-cast risks (M4) make this an error-typing + naming cluster, not pure cosmetics.

## 6. Findings

### Blocker

#### B1: [src/per_sample_caller/baq/engine.rs:121](../../src/per_sample_caller/baq/engine.rs#L121) — `ref_fetcher.fetch` errors silently collapsed into `RefWindowPastChromEnd`
- **Categories:** errors
- **Confidence:** High
- **Problem:**
  ```rust
  let ref_bytes = match ref_fetcher.fetch(read.ref_id as u32, (xb + 1) as u32, length) {
      Ok(bytes) if !bytes.is_empty() => bytes,
      _ => return BaqOutcome::Skipped(BaqSkipReason::RefWindowPastChromEnd),
  };
  ```
  `RefSeqFetcher::fetch` returns `Result<Vec<u8>, std::io::Error>`. The trait's docstring at `pileup/mod.rs:450-462` explicitly states the error covers both *"the range exceeds the chromosome's length"* and *"the fetch itself fails (FASTA I/O)"*. The wildcard arm conflates both — plus the `Ok(empty)` case — into a single skip reason. A transient FASTA read error or a corrupt index is silently dropped and counted as a normal boundary case. There is no `tracing` event, no error chain, and operators reading skip telemetry cannot distinguish a healthy run from a degraded one.
- **Why it matters:** This is data loss masquerading as routine filtering, on a production-bound code path. The category's hard rule is *meaningful `Result`s must not be silently discarded*.
- **Suggested fix:** Split the boundary case from real I/O errors. The minimum acceptable change:
  ```rust
  let ref_bytes = match ref_fetcher.fetch(read.ref_id as u32, (xb + 1) as u32, length) {
      Ok(bytes) if bytes.is_empty() => {
          return BaqOutcome::Skipped(BaqSkipReason::RefWindowPastChromEnd);
      }
      Ok(bytes) => bytes,
      Err(e) => {
          tracing::error!(
              error = ?e, ref_id = read.ref_id, xb, length,
              "BAQ ref fetch failed",
          );
          return BaqOutcome::Skipped(BaqSkipReason::RefWindowPastChromEnd);
      }
  };
  ```
  Preferred long-term: tighten the trait contract so `fetch` returns `Result<Option<Vec<u8>>, std::io::Error>` (`Ok(None)` for past-chrom-end, `Err` reserved for genuine I/O), and add a `BaqOutcome::Failed(BaqEngineError::RefFetch { … })` variant so the stream can propagate the error like `CramInputError`.

#### B2: [src/per_sample_caller/baq/probaln.rs:62](../../src/per_sample_caller/baq/probaln.rs#L62) — `probaln_glocal` input-validation paths are untested
- **Categories:** reliability, extras (convergent)
- **Confidence:** High
- **Problem:** `probaln_glocal` documents four early-return paths (length mismatch among `iqual`/`query`/`state`/`q`; `ref_seq.len()` or `query.len()` not fitting in `i32`; `l_query >= i32::MAX - 2`; `l_ref == 0 || l_query == 0`). None of them has a unit test. The parity fixtures feed only well-formed inputs; `BaqEngine` enforces lengths upstream so the validator is effectively unreachable through the production path. A refactor that removed any of these guards would silently change `probaln_glocal`'s behavior (e.g. `l_ref == 0` would index `ref_seq[0]` and divide `bm` / `bi` by zero, filling `q` with NaNs that the `(... + 0.499) as i32` cast then truncates to `0` — perfectly normal-looking Phred-0 qualities downstream).
- **Why it matters:** Per the severity rubric, "absence of tests for any code path that, if broken, would produce wrong results without panicking" is a Blocker.
- **Suggested fix:** Add direct tests for each branch. Minimal four-test sketch:
  ```rust
  #[test]
  fn probaln_glocal_returns_invalid_input_on_length_mismatch() { /* iqual.len() != query.len() */ }
  #[test]
  fn probaln_glocal_returns_ok_zero_on_empty_query() { /* l_query == 0 → Ok(0) */ }
  #[test]
  fn probaln_glocal_returns_ok_zero_on_empty_ref() { /* l_ref == 0 → Ok(0) */ }
  #[test]
  fn probaln_glocal_returns_invalid_input_on_negative_bandwidth() { /* cfg.bw = -1 */ }
  ```
  Full bodies in `tmp/review_2026-05-12_baq/reliability.md`. Lift `extend_ref_window`, `locate_alignment_span`, `set_u`, and the `Q2P` static to `pub(super)` (already true for `set_u`-class) and add helper-level tests alongside.

### Major

#### M1: [src/per_sample_caller/baq/errors.rs:12](../../src/per_sample_caller/baq/errors.rs#L12) — `BaqOverflow` enum name misleads on its `InvalidInput` variant
- **Categories:** naming, errors (convergent)
- **Confidence:** High
- **Problem:** The enum is named `BaqOverflow`, but only `AllocationOverflow` is an overflow; `InvalidInput` covers length mismatches and `i32`-range violations. A reader who sees `-> Result<_, BaqOverflow>` at a use site assumes every error is a memory/numeric overflow and misses the input-validation failure mode. The user flagged this explicitly.
- **Why it matters:** The type name lies about the variant — the "actively misleads" bar in the naming rubric. The enum is re-exported publicly from `mod.rs:28`.
- **Suggested fix:** Rename to `ProbalnError` (preferred) or `BaqHmmError`:
  ```rust
  // errors.rs
  #[non_exhaustive]                    // see M3
  #[derive(Error, Debug, Clone, Copy, PartialEq, Eq)]
  pub enum ProbalnError {
      #[error("BAQ slice lengths disagree (iqual/state/q must match query)")]
      SliceLengthMismatch,             // see M2 for split
      #[error("BAQ query or reference exceeds i32 range")]
      SequenceTooLong,
      #[error("BAQ matrix allocation would overflow")]
      AllocationOverflow,
  }
  ```
  Update the re-export in `mod.rs:28` and the references in `probaln.rs:17,70,72,77,103,106,109,111` and `engine.rs:48,50`. Mechanical rename; the `BaqSkipReason::HmmOverflow` variant in `engine.rs:50` is fine as-is — it names the *outcome*, not the error.

#### M2: [src/per_sample_caller/baq/errors.rs:12-23](../../src/per_sample_caller/baq/errors.rs#L12-L23) — `BaqOverflow::InvalidInput` lumps three distinct invariants
- **Categories:** errors (convergent with M1 in naming)
- **Confidence:** High
- **Problem:** The single `InvalidInput` variant fires from five sites in `probaln.rs` covering:
  - `iqual.len() != query.len() || state.len() != query.len() || q.len() != query.len()` (engine-side contract violation)
  - `i32::try_from(ref_seq.len())` failure
  - `i32::try_from(query.len())` failure
  - `l_query >= i32::MAX - 2`
  The `Display` message `"BAQ inputs are invalid (length mismatch or out of i32 range)"` hides which one fired. Downstream, `engine.rs:153` maps the whole result to `BaqSkipReason::HmmOverflow`, erasing the distinction. A run with an engine bug looks identical in telemetry to a run with pathological reads.
- **Why it matters:** The length-mismatch case is an *engine bug* (the engine sizes `state`/`q` from `read.seq.len()` and checks `qual.len() == seq.len()` — a violation here is unreachable today but would indicate a regression). The i32-range cases are *data-driven* skips. The two have different operational responses; lumping them erases the signal.
- **Suggested fix:** Split into three variants (see M1 snippet) and either:
  1. Demote the length-mismatch checks to `debug_assert_eq!` since the engine already enforces them, or
  2. Surface the engine-bug case as a `BaqOutcome::Failed` rather than a skip, so it ends up in logs instead of in the per-read skip counter.

#### M3: [src/per_sample_caller/baq/errors.rs:11](../../src/per_sample_caller/baq/errors.rs#L11) — `BaqOverflow` is publicly re-exported but lacks `#[non_exhaustive]`
- **Categories:** errors
- **Confidence:** High
- **Problem:** `BaqOverflow` is exported through `mod.rs:28` and downstream consumers can pattern-match exhaustively on its variants. Adding the variant split from M2 — or any future variant — is a semver-breaking change.
- **Why it matters:** Category rule "Public error types are `#[non_exhaustive]` so variants can be added without a semver break."
- **Suggested fix:** Add `#[non_exhaustive]` (snippet in M1). Same treatment for `BaqSkipReason` if it ever leaves the crate.

#### M4: [src/per_sample_caller/baq/engine.rs:109,114,121](../../src/per_sample_caller/baq/engine.rs#L109) — Silent narrowing casts on user-controlled SAM fields
- **Categories:** errors, reliability, idiomatic, refactor_safety (convergent — four sub-agents flagged this independently)
- **Confidence:** High
- **Problem:** Three narrowing casts on `MappedRead` fields with no range checks:
  - `read.pos as i32 - 1` (line 109) — `pos` is `u64`. A position ≥ 2^31 (long plant/amphibian chroms) truncates and produces a negative `pos_0`. `locate_alignment_span` then walks from a corrupted `x` start.
  - `read.seq.len() as i32` (line 114) — `usize → i32`. The `try_from` inside `probaln_glocal` only guards the *HMM-side* `query` length; the engine has already done arithmetic on `l_qseq` by then.
  - `read.ref_id as u32` (line 121, also line 338) — `usize → u32`. On a 64-bit host, `ref_id > u32::MAX` silently wraps to a bogus chrom id.
- **Why it matters:** `MappedRead` is decoded from external CRAM/SAM; its fields are not type-bounded. Plain `as` casts wrap silently in release. The category rule on "integer arithmetic on untrusted-input values" requires `checked_*` / `try_into`.
- **Suggested fix:** Replace each cast with a checked conversion that produces a structured skip:
  ```rust
  let pos_0 = i32::try_from(read.pos)
      .ok()
      .and_then(|p| p.checked_sub(1))
      .ok_or(BaqSkipReason::PosOutOfRange);  // new variant
  let l_qseq = i32::try_from(read.seq.len())
      .map_err(|_| BaqSkipReason::ReadTooLong)?;  // new variant
  let chrom_id = u32::try_from(read.ref_id)
      .map_err(|_| BaqSkipReason::ChromIdOutOfRange)?;
  ```
  Cheaper alternative: document upstream invariants in `cram_input` (e.g. "pos < i32::MAX after filter") and add `debug_assert!(read.pos < i32::MAX as u64)` here citing that invariant.

#### M5: [src/per_sample_caller/baq/engine.rs:116-117](../../src/per_sample_caller/baq/engine.rs#L116-L117) — `(xe - xb) as u32` wraps if `extend_ref_window` underflows
- **Categories:** errors
- **Confidence:** Medium
- **Assumptions:** `extend_ref_window` can in principle yield `xe < xb` under pathological `bw`; I have not constructed an input that produces it.
- **Problem:** `let length = (xe - xb) as u32;` after `extend_ref_window`. The function applies an asymmetric rebalance (`xb` updated first, `xe -= dx` computed against the *updated* `xb`); the comment-operator order makes underflow possible in extreme bandwidths. The only sanity check is `if length == 0`. A negative difference produces a near-`u32::MAX` length, which is either a 4 GiB fetcher request or a `RefWindowPastChromEnd` skip — both wrong reactions.
- **Why it matters:** Same root cause as M4 — unchecked signed→unsigned narrowing on a derived value.
- **Suggested fix:**
  ```rust
  if xe <= xb {
      return BaqOutcome::Skipped(BaqSkipReason::RefWindowPastChromEnd);
  }
  let length = u32::try_from(xe - xb).expect("xe > xb checked above");
  ```

#### M6: [src/per_sample_caller/baq/engine.rs:311](../../src/per_sample_caller/baq/engine.rs#L311) and [engine.rs:372](../../src/per_sample_caller/baq/engine.rs#L372) — Wildcard `_ => {}` arms in CIGAR walks absorb future `CigarOp` variants
- **Categories:** refactor_safety
- **Confidence:** High
- **Problem:** Three CIGAR walks live in this file with inconsistent exhaustiveness:
  - `locate_alignment_span` (line 214-233) is **exhaustive** — every variant is named, no `_` arm.
  - `apply_baq_cap_into` (line 311) uses `_ => {}` after `Match | SeqMatch | SeqMismatch`, `SoftClip | Insertion`, `Deletion`.
  - `compute_alignment_end` (line 372) uses `_ => {}` after the ref-consuming arm.
  Adding a new `CigarOp` variant — even one that should consume reference or query bases — silently no-ops in the last two walks while triggering a compile error in the first. The result is a misaligned `(x, y)` cursor and silently wrong `bq_baq` for reads containing the new op.
- **Why it matters:** htslib byte-for-byte parity is the module's central invariant. The compiler can enforce "every CIGAR-walk handles every variant" if and only if every walk is exhaustive; today only one is.
- **Suggested fix:** Replace each `_ => {}` with explicit variants. For `apply_baq_cap_into`:
  ```rust
  match *op {
      CigarOp::Match(_) | CigarOp::SeqMatch(_) | CigarOp::SeqMismatch(_) => { … }
      CigarOp::SoftClip(_) | CigarOp::Insertion(_) => { … }
      CigarOp::Deletion(_) => { … }
      CigarOp::HardClip(_) | CigarOp::Padding(_) => {}      // ref+query neither
      CigarOp::Skip(_) => {}                                // rejected upstream; spell out
  }
  ```
  Same shape for `compute_alignment_end`. After the fix, adding a `CigarOp` variant is a compile error at all three walks.

#### M7: [src/per_sample_caller/baq/engine.rs:131](../../src/per_sample_caller/baq/engine.rs#L131) — `BaqConfig { bw, ..self.cfg }` spread silently absorbs new fields
- **Categories:** refactor_safety
- **Confidence:** High
- **Problem:** `let hmm_cfg = BaqConfig { bw, ..self.cfg };` inherits `d` and `e` from `self.cfg` and overrides `bw`. When a new `BaqConfig` field lands (a tuning knob, a parity-affecting tweak), this site silently inherits it from `self.cfg` instead of forcing the author to decide what the per-read HMM sees. Refactor-safety rule "no `..Default::default()`" applies to *any* struct-literal spread — particularly to a struct whose meaning is tied to a parity contract.

#### M8: [src/per_sample_caller/baq/probaln.rs](../../src/per_sample_caller/baq/probaln.rs) (whole file) — htslib-parity invariants have no compile-time guard
- **Categories:** refactor_safety
- **Confidence:** High
- **Problem:** Three classes of silent-drift exposure with no test or lint catching them:
  1. **Magic-literal constants.** `EM = 0.33333333333` (line 25) is pinned by comment ("Match the literal exactly for parity"). A clippy `approx_constant` suggestion to use `1.0/3.0` would compile, pass clippy, and silently break parity on Phred-rounding-boundary inputs. Same risk for `4.343` (Phred conversion, lines 257, 261, 382).
  2. **Float-associativity order.** Lines 198-199 hold `xm3 = EI * m_scale * m[1]` and `xm4 = EI * m_scale * m[4]`, then `f[base_i + u + 1] = xm3 * f[…] + xm4 * f[…]`. The comment block 182-187 explains this is the OPTIMIZED-branch order. A refactor that reassociates (`EI * (m_scale * f[…] * m[1] + …)`) compiles, passes clippy, silently changes rounding.
  3. **`set_u` boundary semantics.** Lines 240, 271, 331 ride on `set_u` returning raw signed arithmetic that wraps to a huge `usize` for out-of-band callers; the boundary check `u < 3 || u >= i_dim` catches it. A "clean up unsigned conversion" refactor that returns `Option<usize>` would compile but skip boundary cells htslib visits.
  The current safety net is two parity fixtures (`realn01`, `realn02`) that exercise a finite set of `(l_ref, l_query, bw)` triples.
- **Why it matters:** The whole point of the BAQ port is parity. The compiler cannot help when an author "modernises" a magic constant.
- **Suggested fix:** Combine these:
  1. Extract magic literals to named consts:
     ```rust
     /// 10/ln(10), 4-digit truncation. Keep — htslib uses the same
     /// truncation, do not promote to `10.0 / f64::ln(10.0)`.
     const PHRED_PER_NAT: f64 = 4.343;
     /// Rescale threshold (probaln.c:283).
     const RESCALE_THRESHOLD: f64 = 1e-100;
     /// Posterior-Q cap. htslib's `if (qv > 100) qv = 99`.
     const PHRED_CAP_OVERFLOW: i32 = 100;
     const PHRED_CAP: u8 = 99;
     ```
  2. Add `#[allow(clippy::approx_constant)]` on `EM` with a doc cross-link.
  3. Add a `// PARITY:` marker at every load-bearing site (lines 25, 198-199, 257). A grep for `PARITY:` then enumerates the contracts a future refactor must respect.
  4. Pin the values in unit tests (`assert!(EM > 0.333333333329 && EM < 0.333333333331)`; same for `PHRED_PER_NAT`). A drift to `1.0/3.0` fails the unit test, not just the parity fixture.

#### M9: [src/per_sample_caller/baq/stream.rs:142-148](../../src/per_sample_caller/baq/stream.rs#L142-L148) — `BaqStream` is not poisoned after a worker panic
- **Categories:** unsafe_concurrency
- **Confidence:** Medium
- **Assumptions:** Reachable only if a downstream consumer wraps iteration in `std::panic::catch_unwind` and continues polling. The pileup walker (the planned consumer) does not yet do this.
- **Problem:** If `engine.process` panics inside `par_drain`, rayon propagates the panic up through `BaqStream::next`. `BaqStream` has no `poisoned` flag — `upstream_done` is still `false`, `pending_error` is still `None`, `current_batch` is empty. A consumer that runs `BaqStream` inside a `catch_unwind` boundary and then calls `next()` will silently begin a new chunk pull, having *lost* every read in the panicking chunk. The walker would see a coordinate-sorted stream with a gap — wrong calls, no error.
- **Why it matters:** Loss of reads silently corrupts variant calls; the missing reads contribute to no pileup column. The walker's "non-decreasing coordinate" invariant is preserved, so nothing downstream notices.
- **Suggested fix:** Add a `poisoned: bool` field and a `Drop`-style guard around `par_drain`:
  ```rust
  struct PoisonGuard<'s>(&'s mut bool);
  impl<'s> Drop for PoisonGuard<'s> {
      fn drop(&mut self) {
          if std::thread::panicking() { *self.0 = true; }
      }
  }
  let _g = PoisonGuard(&mut self.poisoned);
  self.chunk_buf.par_drain(..).map_init(/* ... */).collect_into_vec(&mut self.outcomes_buf);
  std::mem::forget(_g);
  ```
  And in `next`: `if self.poisoned { return None; }` at the top.

#### M10: [src/per_sample_caller/baq/probaln.rs:62-387](../../src/per_sample_caller/baq/probaln.rs#L62-L387) — `probaln_glocal` is one ~325-line function with four labelled phases
- **Categories:** smells
- **Confidence:** High
- **Problem:** Forward / likelihood / backward / MAP-decode all live in one body, sectioned only by `// === Forward ===` banners. ~14 mutable locals cross phase boundaries; each phase has its own bounds-elision asserts and `set_u` hoisting. The parity-with-htslib constraint pins the *expressions*, not their packaging.

## 9. What's good

- **Parity-comment density.** Every load-bearing choice that mirrors htslib has a comment with a file:line reference (`probaln.rs:21-25` on `EM`, `probaln.rs:181-187` on the OPT-branch reassociation, `engine.rs:246-248` on the comma-operator rebalance). New contributors get the *why* without reading htslib.
- **`mapped_to_prepared` spells every `PreparedRead` field explicitly.** The deliberate non-`#[non_exhaustive]` policy on `PreparedRead` plus the explicit literal in [engine.rs:337-349](../../src/per_sample_caller/baq/engine.rs#L337-L349) means a new `PreparedRead` field is a compile error here — the right shape for this struct's intent.
- **`BaqSkipCounts::bump` is exhaustively named per-reason** ([stream.rs:42-52](../../src/per_sample_caller/baq/stream.rs#L42-L52)), so a new `BaqSkipReason` variant forces a compile error here. The right reference style for the rest of the module's matches.
- **`map_init` per-worker engine pattern** ([stream.rs:144-146](../../src/per_sample_caller/baq/stream.rs#L144-L146)) — one `BaqEngine` per rayon worker, never shared, scratch reused across reads on that worker. Clean isolation, no `Sync` on the engine.
- **The dispatch table optimisation for `encode_base`** ([engine.rs:180-191](../../src/per_sample_caller/baq/engine.rs#L180-L191)) — a 256-byte `static` LUT instead of a 5-way `match` enables `Vec::extend(iter.map(encode_base))` to vectorize on AVX2/NEON, and the change is local + documented.
