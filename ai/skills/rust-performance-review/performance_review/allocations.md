# Allocations & memory pressure checklist

**Purpose.** Identify hot-path code that allocates per call, clones owned data unnecessarily, or repeatedly grows containers ‚Äî the dominant source of preventable cost in idiomatic Rust.

**Triggers.** Search the in-scope code for: `Vec::new`, `String::new`, `vec![]`, `format!`, `.to_owned()`, `.to_string()`, `.clone()`, `Box::new`, `Arc::new`, `HashMap::new`, `BTreeMap::new`, any owning type used as a function argument where a borrow would do, any iterator chain that ends in `.collect()` only to be iterated again.

**Skip when.** Scope is purely cold code (CLI parsing, startup, error reporting). The orchestrator should not dispatch this category for snippets that obviously do not allocate; if dispatched anyway, write `No findings.`

## Rules

- **Allocations belong outside hot loops.** A `Vec` / `String` / `HashMap` constructed inside a per-iteration loop is a finding. Hoist it above the loop and call `.clear()` between iterations ‚Äî `clear` keeps capacity. Mechanism: turn N allocations into 1.
- **Pre-size containers when the size is known or bounded.** `Vec::with_capacity(n)` and `String::with_capacity(n)` cost one allocation; `push` into a default `Vec` costs O(log n) reallocations (capacities 0 ‚Üí 4 ‚Üí 8 ‚Üí 16 ‚Üí ‚Ä¶). Filed when a `Vec::new()` is followed by a known-bounded number of pushes.
- **`.collect::<Vec<_>>()` followed by another iteration is wasted work.** Either chain the iterators, or use `.collect_into()` on a reused `Vec`. Filed for any `let v: Vec<_> = iter.collect(); v.iter().for_each(...)` that does not mutate `v` between the two.
- **Clones are evidence, not noise.** Each `.clone()` on a `Vec`, `String`, `Arc`, or non-`Copy` struct in hot code requires a one-line justification. Patterns that produce findings: `vec.iter().map(|x| x.clone())` that the body could borrow; passing `String` by value when the callee only reads; cloning `Arc<T>` to share a single read access (use a borrow). `Arc::clone` is one atomic increment ‚Äî cheap, not free.
- **Replace big-clone-per-comparison with cached projections.** When a sort or filter calls a function that clones a large struct on every call, project the fields the comparison reads into a small `RowCache` struct, refresh once per update, and run the loop over the cache. See `data_layout.md`.
- **`String` / `Vec<T>` parameters where `&str` / `&[T]` work are findings.** Same for `PathBuf` / `&Path`, `Box<T>` / `&T`. Per-call ownership transfers force per-call allocations on every caller. Use `impl AsRef<Path>` / `impl AsRef<str>` for builders.
- **`format!` in a hot path is an allocation.** Replace with `write!(buffer, ...)` against a reused `String` or `Vec<u8>`. Filed for any `format!` whose output is immediately appended to another `String` or used inside a per-record loop. The choice between `format!`, `write!`, and `Display::fmt` overlaps with `hot_loops.md` ‚Äî file here when the cost is the allocation, there when the cost is per-byte formatting overhead.
- **Tiny, often-empty collections are SmallVec candidates ‚Äî sometimes.** `SmallVec<[T; N]>` keeps up to N items inline; for collections that are usually small (1‚Äì8), this turns N heap touches into N stack writes. Caveat: SmallVec is *slightly slower* than `Vec` for the heap-allocated case because every operation checks inline-vs-heap. File at Likely with the experiment "swap to SmallVec, rerun bench, compare".
- **Arena allocators win when many short-lived objects share a phase.** Per-record temporary state, parser ASTs, per-pipeline-stage scratch ‚Äî `bumpalo` use cases. Bump-allocate, drop the arena at end of phase, no individual frees. Trade-off: every object lives until the arena drops, so an arena that holds across phases leaks. File at Likely when the lifetime structure is clean.
- **Interning beats repeated string allocation.** If the same N strings show up M ‚â´ N times in keys / values (chromosome names, sample IDs, contig keys), an interning crate (`string-cache`, `lasso`, `internment`) flattens M allocations into N + cheap copies of an integer handle.
- **`Box<dyn Trait>` collected into a `Vec` is two allocations per element** (one for the object, one for the vec slot ‚Äî and the vec stores fat pointers, so it is a pointer-chase per access). If the concrete types form a closed set, an `enum` of variants stores inline. File only when the enum is small and dispatch overhead is not the actual bottleneck. Cross-reference: `hot_loops.md` on static vs dynamic dispatch.
- **Watch for accidental `Cow::Owned`.** `Cow<'_, str>` that always ends up `Owned` is a `String` with extra ceremony.
- **Global allocator choice is a build configuration finding.** A workload that allocates heavily and ships with the system allocator without an A/B benchmark is filed in `methodology.md` as a build-config finding, not here. Mention it as a cross-category observation if you spot it.
